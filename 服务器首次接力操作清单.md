# 服务器首次接力操作清单

## 1. 这份文档解决什么问题

这份文档专门解决下面 4 个现实问题：

1. 现在 GitHub 上这份基线，够不够你切到服务器继续干活。
2. 哪些东西已经上传了，哪些东西本来就不应该上传。
3. `Trae`、`PyCharm`、`Git` 三者到底怎么分工才不会互相打架。
4. 服务器第一次接力时，应该按什么顺序准备环境、放数据、开始正式工作。

这份文档不是讲抽象原理，而是给你后面连服务器时直接照着执行。

## 2. 当前结论先说清楚

先直接说结论：

**当前 GitHub 上的内容，已经足够作为服务器接力的正式起点。**

但要同时明确：

**它不是整个项目的“全部资产全集”，而是“适合版本管理、适合跨机器接力的正式基线”。**

也就是说，当前已经上传的内容，足够你在服务器上继续：

- 拉取代码
- 拉取配置
- 拉取阶段文档
- 拉取门禁说明
- 拉取关键实验元数据
- 拉取接力说明

但当前不会通过 GitHub 同步这些内容：

- 数据集
- checkpoint
- 大量可视化图片
- 大量预测结果
- 大型中间缓存
- 生成型报告资产

这不是缺陷，而是正常设计。

## 4. 当前 GitHub 基线里故意没有什么

下面这些内容没有放进 GitHub，而且后面也不应该指望它们通过 Git 自动同步：

- `datasets/`
- `**/checkpoints/`
- `**/*.ckpt`
- `**/visuals/`
- `**/predictions/`
- `crc_gland_segmentation_project/reports/data_preview/`
- `crc_gland_segmentation_project/reports/data_targets/`
- 大体积参考资料库
- 历史归档目录
- 大型抽取缓存

原因很简单：

1. 这些东西太大，不适合 GitHub 做主同步通道。
2. 这些东西经常变化，进 Git 只会让仓库臃肿和冲突失控。
3. 你真正需要跨窗口、跨机器稳定传递的是“代码 + 配置 + 文档 + 关键元数据”，不是所有大文件。

## 5. 服务器第一次接力时到底够不够

如果你的目标是下面这些事，那么当前 GitHub 内容已经够了：

- 在服务器上 clone 项目
- 恢复当前阶段上下文
- 检查训练入口
- 检查 CUDA / PyTorch / AMP
- 修改代码
- 修改阶段文档
- 回填实验元数据
- 提交和推送轻量结果说明

如果你的目标是下面这些事，那么 GitHub 内容本身还不够，需要额外准备：

- 在服务器上拿到数据集
- 在服务器上放 checkpoint
- 在服务器上保留大量图片结果
- 把大批量预测结果同步回本地

这类内容必须走别的通道，例如：

- 服务器本地磁盘
- 云盘 / 网盘
- SFTP
- PyCharm 的远程下载
- 手动打包下载

## 6. 推荐总分工

后面最稳的方案不是只靠一种工具，而是三套东西分工明确：

### 6.1 Git 负责什么

`Git / GitHub` 只负责：

- 代码
- 配置
- 根目录说明文档
- 阶段状态页
- 关键小型实验元数据
- 接力文档

一句话：

**Git 负责“项目状态的正式文本基线”。**

### 6.2 Trae 负责什么

`Trae` 更适合负责：

- 在服务器上执行命令
- 跑检查
- 跑训练
- 跑评估
- 直接修改服务器上的代码与文档
- 快速读取项目上下文并推进任务

一句话：

**Trae 负责“服务器上的执行与决策推进”。**

### 6.3 PyCharm 负责什么

`PyCharm` 更适合负责：

- 远程浏览服务器工程
- 看日志
- 搜文件
- 查看远程解释器状态
- 通过 Deployment / SFTP 下载部分结果
- 做本地阅读和辅助编辑

一句话：

**PyCharm 负责“工程浏览、远程开发辅助和按需下载”。**

---

## 7. 你说的“Trae 连服务器 + PyCharm 同时连服务器”能不能用

可以用。

**但前提是分工必须非常清楚。**

最推荐的现实做法是：

- `Trae` 作为服务器主操作入口
- `PyCharm` 作为远程浏览和辅助入口
- `GitHub` 作为正式版本留痕和轻量同步入口

这三者可以同时存在，但不要让它们同时做同一件事。

### 推荐分工

- `Trae`：主要改服务器上的代码、文档、脚本，并执行命令
- `PyCharm`：主要看远程目录、看日志、按需下载结果、做少量辅助编辑
- `Git`：主要同步文本类工程资产，不负责承载大结果

### 不推荐做法

- `Trae` 正在改某个文件时，`PyCharm` 又同步覆盖同一个文件
- 本地 `PyCharm` 自动上传整个项目去覆盖服务器
- 服务器改了一堆内容，但不 commit，不写接力文档，只靠记忆继续

---

## 8. 最重要的一条边界

你后面完全可以把 Git 当“保险 + 正式留痕通道”。

这个理解是对的。

但要补一句：

**Git 虽然不负责同步所有大文件，但它必须负责同步所有“决定项目状态”的关键文本资产。**

也就是说，下面这些即使很小，也一定要走 Git：

- 代码
- 配置
- 实验说明文
- 阶段状态页
- 接力文档
- 关键元数据
- 小型总结表

如果这些不走 Git，只靠服务器文件本身，那一旦：

- 开新窗口
- 换机器
- 误覆盖
- 误删除

就会丢掉真正最重要的状态。

## 9. 大资产后面怎么处理

后面服务器上真正会不断变多的大资产，一般包括：

- 数据集
- checkpoint
- tensorboard 日志
- predictions
- visuals
- 大批量 csv
- 批量错误案例图

这些资产推荐分三类管理。

### 9.1 必须长期保留但不需要本地实时同步

例如：

- 正式训练 checkpoint
- 大批量预测结果
- 大量可视化图

推荐做法：

- 主要保留在服务器
- 按阶段打包
- 只在真正需要时下载到本地

### 9.2 需要本地查看但不需要进 Git

例如：

- 少量关键结果图
- 若干错误案例
- 某次 run 的日志导出

推荐做法：

- 通过 `PyCharm Deployment` 或 SFTP 单独下载
- 下载到本地单独结果目录
- 不重新加入 Git

### 9.3 必须进入 Git 的轻量关键信息

例如：

- `run_meta.yaml`
- `run_summary.md`
- 关键指标汇总表
- 状态页更新
- 说明文更新

这类必须 commit + push。

## 10. 服务器首次接力时的执行顺序

下面是最推荐的第一次服务器接力顺序。

### 第 1 步：登录服务器后先别急着跑训练

先确认这些基础项：

- `nvidia-smi`
- `python --version`
- `git --version`
- `nvcc --version` 或确认 CUDA 可用

### 第 2 步：clone 当前 GitHub 基线

执行：

```bash
git clone https://github.com/hyc-keep/Paper.git
cd Paper
```

### 第 3 步：进入项目后先读接力文档

优先读取：

1. `Paper/服务器切换接力说明.md`
2. `Paper/服务器首次接力操作清单.md`
3. `Paper/02_UNet流程验证_执行检查清单.md`

### 第 4 步：准备服务器本地的大资产目录

在服务器上准备这些目录，但不要把它们重新纳入 Git：

- `datasets/`
- `crc_gland_segmentation_project/datasets/`
- `crc_gland_segmentation_project/experiments/.../checkpoints/`
- `crc_gland_segmentation_project/experiments/.../visuals/`
- `crc_gland_segmentation_project/experiments/.../predictions/`

### 第 5 步：把数据集放到服务器

这一步不要依赖 Git。

数据集应通过以下方式之一进入服务器：

- 服务器已有挂载盘
- 手动上传
- SFTP
- 云盘中转

### 第 6 步：验证 PyTorch 和 CUDA

至少确认下面几点：

- `torch.cuda.is_available() == True`
- 可以识别到目标 GPU
- 显存信息正常
- 训练入口能加载配置与数据

### 第 7 步：先做 GPU smoke check

先不要直接把服务器当成正式结论生产线。

先做：

- forward
- loss
- backward
- optimizer.step
- AMP 路径检查

确认都正常后，再进入正式 A1。

### 第 8 步：正式 run 期间以服务器为主线

从这一刻开始，建议规则固定为：

- 正式代码修改优先在服务器
- 正式文档回填优先在服务器
- 正式 run 结果记录优先在服务器

### 第 9 步：重要文本资产及时 push

每完成一段稳定工作，就把下面这些推上 GitHub：

- 新增或修改的代码
- 配置变更
- 阶段状态页
- 接力文档
- 关键元数据

### 第 10 步：需要本地查看的大结果再单独下载

不要为了同步几张图、几个 checkpoint，就把仓库结构重新搞乱。

大结果只做按需下载，不做全量 Git 同步。

## 11. Trae 和 PyCharm 同时连接时的硬规则

为了避免服务器内容被自己人互相覆盖，后面请尽量守这几条：

1. 同一时间只让一个工具改同一个文件。
2. `Trae` 正在改代码时，不要让 `PyCharm` 执行自动上传覆盖。
3. `PyCharm` 更适合下载结果、看日志、搜文件，不适合替代 Git 做全工程同步。
4. 本地如果只是想“同步服务器最新代码”，优先 `git pull`，不是拖拽复制。
5. 大文件同步用 SFTP / 手动下载，不要硬塞进 Git。

## 13. 最后一句话

后面真正稳定的工作流不是“只靠 Git”，也不是“只靠 PyCharm 同步”，而是：

**Git 管关键文本状态，Trae 管服务器执行，PyCharm 管远程浏览和按需下载，大资产留在服务器。**

只要这四句话不乱，后面服务器即使改很多东西，整体也不会失控。
