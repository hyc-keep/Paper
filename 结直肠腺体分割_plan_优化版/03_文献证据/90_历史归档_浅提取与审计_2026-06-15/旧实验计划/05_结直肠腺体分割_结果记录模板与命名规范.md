# 结直肠腺体分割结果记录模板与命名规范

这份文档专门解决结果管理问题。

但从科研 protocol 角度看，它真正解决的是另一件事：

> 怎样把每一轮实验保存成“可复查证据”，而不是保存成“看起来还行的结果”。

它回答下面这些问题：

1. 每次实验 run 应该怎么命名
2. 每次 run 最少要保存哪些文件
3. `summary.md` 和 `error_cases.md` 该怎么写
4. 表和图该怎么编号
5. 什么样的结果才算“正式结果”

如果没有这份文档，后面最容易出现的问题不是“文件丢了”，而是：

- 你不知道某个结论到底基于哪个协议
- 你不知道某个模块到底和上一版只差哪一个变量
- 你不知道某个结果能不能写进正文主表

---

## 0. 这份文档的科学定位

### 0.1 记录不是行政动作，而是证据固定

腺体分割这类实验里，后面要写进论文的不是：

- 一个最高分
- 一张最好看的图
- 一次临时跑出来的终端输出

真正能支撑论文结论的是：

- 固定协议下的 run 记录
- 多 seed 后的 `mean +- std`
- 对象级和边界级指标
- 失败案例
- 与上一版唯一变量对应的比较说明

### 0.2 直接论文依据

这份文档虽然不对应某一篇具体方法论文，但它受下面几类文献结论约束：

- `GlaS Challenge` (GlaS_Challenge_Contest_2017.pdf, p.8, p.9, p.10)
  - 提醒你对象级与边界级指标必须能追溯
- `Current Trends of Artificial Intelligence for Colorectal Cancer Pathology Image Analysis` (Current_Trends_of_Artificial_Intelligence_for_Colorectal_Cancer_Pathology_Image_Analysis_A_Systematic_Review_2020.pdf, p.4)
  - 提醒你腺体分割不能只报单一指标
- `nnU-Net` (nnU-Net_a_self-configuring_method_for_deep_learning-based_biomedical_image_segmentation_2021.pdf, p.3)
  - 提醒你 pipeline 和协议记录本身就是结果可信度的一部分
  - **参数来源类型：论文直接固定**

## 1. run 命名统一规则

当前统一命名格式：

```text
[阶段]_[模型]_[数据集]_[版本]_[seed]
```

示例：

- `A1_UNet_GlaS_v1_seed3407`
- `B2_UNet_GlaS_v1_seed1234`
- `C3_R34UNet_GlaS_v1_seed2025`
- `D1_R34UNet_LKMA_GlaS_posB_k15_seed3407`
- `E1_R34UNet_LKMA_Boundary_GlaS_w3_l03_seed3407`
- `G1_UNetPP_GlaS_v1_seed3407`
- `G2_DeepLabV3P_GlaS_v1_seed3407`
- `G3_AttUNet_GlaS_v1_seed3407`
- `H1_Final_CRAG_v1_seed3407`

不允许使用：

- `new_run`
- `final2`
- `best_try`
- `good_result`
- `last_last`

## 2. 每个正式 run 的目录结构

最少结构：

```text
experiments/
└─ [run_name]/
   ├─ config.yaml
   ├─ seed_record.txt
   ├─ training_log.csv
   ├─ val_metrics.csv
   ├─ testA_metrics.csv / testB_metrics.csv 或 test40_metrics.csv
   ├─ summary.md
   ├─ error_cases.md
   ├─ checkpoints/
   │  └─ best_checkpoint.pth
   ├─ predictions/
   └─ visualizations/
```

如果少于这些文件，默认不算完整 run。

## 3. `summary.md` 该怎么写

每个 run 的 `summary.md` 最少写下面内容：

```md
# Run Summary

- run_name:
- model:
- dataset:
- split_version:
- seed:
- main_change:
- best_epoch:
- best_val_metric:
- testA_summary:
- testB_summary:
- qualitative_observation:
- current_judgment: keep / unclear / drop
```

### 3.1 为什么必须写这些字段

这些字段不是为了凑模板，而是分别对应一个科学问题：

- `main_change`
  - 这轮到底只改了什么变量
- `best_val_metric`
  - 最佳 checkpoint 是按什么规则选出来的
- `testA_summary / testB_summary`
  - 结果是否跨 split 一致
- `qualitative_observation`
  - 数字变化是否有可视化支持
- `current_judgment`
  - 当前版本是保留、待观察还是删除

### 3.2 合格写法示例

```md
# Run Summary

- run_name: C1_R34UNet_GlaS_v1_seed3407
- model: ResNet34-U-Net
- dataset: GlaS
- split_version: glas_train68_val17_testA60_testB20_v1
- seed: 3407
- main_change: only replace UNet encoder with ResNet34 encoder
- best_epoch: 54
- best_val_metric: Dice 0.842
- testA_summary: Dice stable, adhesion cases slightly improved
- testB_summary: malignant hard cases still under-segmented
- qualitative_observation: boundaries are smoother than UNet baseline
- current_judgment: keep
```

### 3.3 不合格写法

不要写：

- `效果不错`
- `感觉变强了`
- `这个可以`

必须写清：

- 变了什么
- 哪个 split 上表现怎样
- 当前判断是保留、待观察还是删除

## 4. `error_cases.md` 该怎么写

每个正式 run 都必须写。

最小模板如下：

```md
# Error Cases

## 1. Main Error Types
- adhesion
- under-segmentation
- over-segmentation
- missing small glands
- boundary blur

## 2. Typical Samples
- sample_id:
  - error_type:
  - short_note:

## 3. Comparison With Previous Run
- better_than_previous:
- unchanged:
- worse_than_previous:

## 4. Current Judgment
- should_keep_current_change:
- reason:
```

### 4.1 为什么 `error_cases.md` 是必须项

腺体分割后面要讲的不是“整体 Dice 高了一点”，而是：

- 黏连有没有分开
- 小腺体有没有漏得更少
- 边界是不是更稳定

所以如果没有失败案例文档，后面你无法科学回答：

- `LKMA` 到底改善了什么
- `Boundary Head` 到底有没有真的帮助分离
- `Distance-aware Loss` 到底是不是只在个别样本上好看

### 4.2 当前必须会分的错误类型

至少分成下面五类：

- 黏连没分开
- 小腺体漏检
- 腺体被切碎
- 边界外扩
- 边界内缩

## 5. 每个阶段必须有的汇总文件

每个阶段结束后，必须汇总：

- `stage_summary.md`
- `mean_std_table.csv`
- `compare_table.csv`

如果阶段涉及消融，还必须有：

- `ablation_table.csv`

如果阶段涉及第一批或第二批外部对比，还必须有：

- `external_protocol_table.csv`
- `external_main_result_table.csv`
- `external_reproduction_note.md`

如果阶段涉及 `CRAG` 补充验证，还必须有：

- `crag_main_result_table.csv`
- `glas_crag_joint_table.csv`
- `crag_stage_summary.md`

### 5.1 为什么这些阶段汇总文件不能少

这些文件各自承担不同证据职责：

- `mean_std_table.csv`
  - 证明不是单次幸运结果
- `compare_table.csv`
  - 证明当前版本只和正确上一版相比
- `ablation_table.csv`
  - 证明模块保留/删除是逐层判断出来的
- `external_protocol_table.csv`
  - 证明外部对比是公平的
- `crag_stage_summary.md`
  - 证明 `CRAG` 的角色是补充验证还是正文支撑

## 8. 什么样的结果才算正式结果

下面全部满足，才算正式结果：

- 有 `config.yaml`
- 有 `training_log.csv`
- 有验证和测试指标文件
- 有最佳 checkpoint
- 有 `summary.md`
- 有 `error_cases.md`
- 有一组可视化图

如果你只有：

- 截图
- 一个最好看的数字
- 一个临时终端输出

那不算正式结果。

### 8.1 什么样的结果才算“可写进论文的结果”

只有下面两类结果，才可以进入论文主表或正文强结论：

1. 固定协议下的单次筛查结果
   - 只能用于首轮趋势判断
   - 不能直接下稳定性结论
2. 固定协议下的多 seed 结果
   - 可用于正式比较、模块保留与否、外部方法胜负判断

换句话说：

- `single_seed` 主要服务筛查
- `mean +- std` 主要服务结论

## 9. `stage_summary.md` 最小模板

每个阶段结束统一写：

```md
# Stage Summary

## 1. Stage
- stage_name:
- target:

## 2. Fixed Protocol
- dataset:
- split:
- input_size:
- loss:
- threshold:
- seeds:

## 3. Main Results
- best_result:
- mean_std:
- main_metric_change:

## 4. Qualitative Findings
- finding_1:
- finding_2:

## 5. Risks
- risk_1:
- risk_2:

## 6. Decision
- pass / not pass
- next_stage:
- keep_or_drop:
```

### 9.1 `stage_summary.md` 必须回答的科学问题

每个阶段结束后，这份文件至少要回答：

1. 当前阶段验证的研究问题是什么
2. 当前阶段唯一变化的变量是什么
3. 结果是否真的支持保留当前改动
4. 如果不过线，问题来自协议、实现还是方法本身

## 10. 什么时候必须报告 `mean +- std`

下面这些情况不能只报单次最好值：

- `UNet` 三次重复
- `ResNet34-U-Net` 三次重复
- `LKMA / Boundary Head / Distance-aware Loss` 的正式保留结论
- 第一批外部对比中和最终方法差距 `<= 1.5` 个百分点的方法
- 准备写入正文主表的 `CRAG` 结果

只允许先报单次 `seed3407` 的情况：

- 第一批外部方法的首轮筛查
- `CRAG` 的最低补充验证首轮检查
- `nnU-Net / TransUNet` 的增强阶段首轮摸底

### 10.1 为什么这里必须严格区分

因为当前项目后面最容易犯的错就是：

- 用单次最好值替代稳定结论
- 用首轮筛查结果假装正式结论

所以这里必须把“筛查结果”和“正式结果”严格分开。

## 11. 外部对比记录时必须额外写什么

每个外部方法的 `summary.md` 除了常规字段，还必须补：

- `implementation_source`
- `paper_setting_gap`
- `fairness_adjustment`
- `whether_need_three_seeds`
- `final_reporting_style: single_seed / mean_std`

`external_reproduction_note.md` 必须明确写：

- 代码来自哪里
- 改了哪些输入输出适配
- 哪些地方没有完全复刻原论文
- 为什么当前比较仍然公平

## 12. CRAG 记录时必须额外写什么

每个 `CRAG` run 的 `summary.md` 还必须补：

- `glas_reference_run`
- `crag_threshold_source`
- `trend_vs_glas: consistent / partly_consistent / inconsistent`
- `trend_vs_crag_baseline: better / close / worse`
- `can_use_in_main_text: yes / no`

`crag_stage_summary.md` 必须写清：

- 最低必跑模型是否已齐：`UNet / ResNet34-U-Net / 最终主方法`
- 是否补了最强外部方法
- 当前结果适合写正文主表还是只适合写补充验证

## 13. 当前最常见的记录错误

### 10.1 没有配置文件

没有 `config.yaml`，后面无法复查这轮到底怎么跑的。

### 10.2 只有最好结果，没有均值方差

没有 `mean +- std`，这轮结论默认不稳。

### 10.3 只有数字，没有图

腺体任务如果没有局部图和失败案例，很难支撑边界和对象分离结论。

### 10.4 只有图，没有错误总结

后面写讨论和实验分析时会非常被动。

### 10.5 只有结果，没有变量说明

如果你只记录“这一版更好”，但没记录和上一版到底只差哪一个变量，那么这轮实验在科研上就不可用。

---

## 14. 你如果只记一句话

> 每一轮实验都必须像“可复查资产”一样保存：有配置、有日志、有指标、有图、有错误记录，缺任何一项都不算正式结果。
