## 1. 先把当前项目说死

### 1.1 当前项目不是在探索方向

到现在为止，方向已经不是“再看看还能做什么”，而是下面这条已经锁定的主线：

- 任务：`2D` 结直肠组织病理腺体分割
- 主数据集：`GlaS`
- 第二 benchmark：`CRAG`
- 流程排雷模型：`UNet`
- 主 baseline：`ResNet34-U-Net`
- 主方法顺序：`+ LKMA -> + Boundary Head -> + Distance-aware Loss(可选)`

这条顺序不是临时想出来的，而是由三条论文证据链共同收缩出来的：

1. 数据与任务链：
   - `GlaS Challenge`：给出 `GlaS` 官方划分与 `F1 / Object Dice / Object Hausdorff` 评价协议（`GlaS_Challenge_Contest_2017.pdf`, p.4, p.8-p.10）
   - `MILD-Net`：给出 `GlaS + CRAG` 双数据集证据链与训练集内部留评估集的做法（`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.2, p.9, p.12）
   - `Current Trends of Artificial Intelligence for Colorectal Cancer Pathology Image Analysis`：说明腺体分割长期围绕 `GlaS / CRAG` 和对象级指标展开（`Current_Trends_of_Artificial_Intelligence_for_Colorectal_Cancer_Pathology_Image_Analysis_A_Systematic_Review_2020.pdf`, p.4）
2. baseline 与对比链：
   - `U-Net`：最小可复核 baseline（`U-Net_2015.pdf`, p.1-p.2）
   - `Deep Residual Learning for Image Recognition`：`ResNet34` 编码器来源（`Deep_Residual_Learning_for_Image_Recognition_2016.pdf`, p.2-p.5）
   - `UNet++ / DeepLabV3+ / Attention U-Net / nnU-Net / TransUNet`：外部对比与公平基线来源，其中 `UNet++` 还直接支持 `BCE + Dice` 组合（`UNet++_2018.pdf`, p.5）
3. 模块动机链：
   - `Large Kernel Matters`：给出大核与有效感受野的最明确数值锚点 `k=15`（`Large_Kernel_Matters_GCN_2017.pdf`, p.5）
   - `Gated-SCNN / DCAN / TA-Net`：说明边界流、轮廓监督和黏连腺体分离不是拍脑袋加的（`Gated-SCNN_Gated_Shape_CNNs_for_Semantic_Segmentation_2019.pdf`, p.1-p.4；`DCAN_Deep_Contour-Aware_Networks_for_Accurate_Gland_Segmentation_2016.pdf`, p.1-p.5；`TA-Net_Topology-Aware_Network_for_Gland_Segmentation_2022.pdf`, p.1-p.5）
   - `Distance Map Loss Penalty Term`：说明边界附近错误值得被额外惩罚（`Distance_Map_Loss_Penalty_Term_for_Semantic_Segmentation_2019.pdf`, p.1-p.5）

这三条证据链在执行层里会继续被分成三种来源类型：

- `论文直接固定`
- `论文支持的候选范围`
- `工程默认起点`

后面只要是参数或协议没有页码锚点，就必须明确写成后两类之一，不能伪装成“论文已经证明的唯一值”。

也就是说，后面真正决定你能不能做成一篇稿子的，不是再找新故事，而是：

- 数据协议是否固定
- `UNet` 是否跑通且跑稳
- 主 baseline 是否可信
- 每个模块是不是按统一协议逐层验证
- 最终表图和失败案例是否齐

### 1.2 当前项目不是在写代码阶段

你现在还没有进入“随手写模型代码”的阶段。

你真正的第一阶段，是把实验地基做完：

1. 数据固定
2. split 固定
3. 检查文件写完
4. `UNet` 首版跑通
5. `UNet` 三次重复稳定

这个顺序同样不是为了“保守”，而是为了满足科学实验的最小要求：

- 先固定数据协议，避免后面每一轮结果不公平
- 先得到 `UNet` 最小流程基线，确认不是数据链或评估链的问题
- 先得到主 baseline，再去判断模块收益来自哪里
- 最后才进入外部对比和第二 benchmark

在这之前，不要把主要精力放到：

- 想 `LKMA` 细节
- 想边界分支结构
- 想 distance map 公式
- 想投稿标题怎么写

### 1.3 当前项目默认目标

后面实验组织默认按下面目标推进：

- 第一目标：先完成一套保底可投稿核心实验包：
  - `UNet` 三次重复
  - `ResNet34-U-Net` 三次重复
  - `+ LKMA`
  - `+ Boundary Head`
  - `Distance-aware Loss` 的保留/删除结论
  - 第一批外部对比：`UNet++ / DeepLabV3+ / Attention U-Net`
  - `CRAG` 补充验证：`UNet / ResNet34-U-Net / 最终主方法`
- 第二目标：如果第一目标做完且趋势稳定，再补投稿增强项：
  - `nnU-Net`
  - `TransUNet`
  - `CRAG` 上补 `1` 个最强外部方法
  - 轻量参数敏感性分析

### 3.2 今天具体只做这 6 件事

#### 第 1 件事：确认当前数据根目录

- 是

#### 第 3 件事：把 CRAG 的当前执行协议想清楚

## 4. 你这周到底怎么安排

### 4.1 第 1-2 天：数据固定

目标：

- 把 `GlaS / CRAG` 数据协议完全固定

必须产出：

- `7` 个 split CSV
- `5` 个检查文件

过线标准：

- 所有 split 文件存在
- 路径真实可读
- image / mask 一一对应
- 你能解释两个数据集当前来源口径

这一步的科学意义：

- 它解决的是协议可复现性，不是模型精度
- 如果这一步没做完，后面所有指标都没有稳定比较意义

### 4.2 第 3-4 天：UNet 首版跑通

目标：

- 跑通训练、验证、测试、可视化

必须产出：

- `config.yaml`
- `training_log.csv`
- `val_metrics.csv`
- `testA_metrics.csv`
- `testB_metrics.csv`
- `best_checkpoint.pth`
- 一组预测图

过线标准：

- 没有大面积全背景
- 没有明显系统性错位
- 有测试结果
- 有可解释的错误类型

这一步的科学意义：

- `UNet` 是最小流程基线，不是最终主模型
- 只要它没跑通，你就不能判断后面任何模块到底有没有用

### 4.3 第 5-7 天：UNet 三次重复

目标：

- 确认当前协议下 `UNet` 稳定可复现

必须产出：

- 3 个 seed 的结果
- `mean +- std`
- 失败案例总结

过线标准：

- 3 次都能跑通
- 波动不离谱
- 失败模式大致一致

这一步的科学意义：

- 这一阶段是在测“协议稳定性”
- 不是在追求最好分数
- 没有这一步，后面主 baseline 和模块消融都会缺少波动参照

只有完成这一步，才有资格进入主 baseline。

### 4.4 第 2 周：主 baseline 与内部模块链

目标：

- 先形成正式比较基座，再逐层验证模块

固定顺序：

1. `ResNet34-U-Net` 三次重复
2. `ResNet34-U-Net + LKMA`
3. `ResNet34-U-Net + LKMA + Boundary Head`
4. `ResNet34-U-Net + LKMA + Boundary Head + Distance-aware Loss`

必须记住：

- 一次只允许改一个变量
- `Distance-aware Loss` 默认是可删项，不是默认必须保留
- 如果 `ResNet34-U-Net` 自己都不稳，不准进入 `LKMA`

这里每一步对应的论文证据是：

- `ResNet34-U-Net`：`ResNet` + `U-Net`
- `LKMA`：`Large Kernel Matters` 等大感受野文献
- `Boundary Head`：`DCAN / Gated-SCNN / TA-Net`
- `Distance-aware Loss`：边界距离惩罚类文献

### 4.5 第 3 周：第一批外部对比与 CRAG 补充验证

目标：

- 把投稿最关键的“外部比较证据”和“第二 benchmark 证据”补齐

第一批外部对比固定只跑：

1. `UNet++`
2. `DeepLabV3+`
3. `Attention U-Net`

`CRAG` 最低必须跑：

1. `UNet`
2. `ResNet34-U-Net`
3. 最终主方法

执行规则：

- 第一批外部方法先跑 `seed3407`
- 如果最终方法和某个外部方法差距 `<= 1.5` 个百分点，或者该方法是主表第二强，就把它补成 `3407 / 1234 / 2025`
- `CRAG` 默认先跑 `seed3407`
- 如果你准备把 `CRAG` 写进正文主表，或者主方法相对 baseline 的优势不明确，就把 `CRAG` 关键模型补成三次重复

这样安排的科学原因是：

- 第一批外部对比首先服务 `GlaS` 主表
- `CRAG` 服务第二 benchmark 补充验证
- 单次 `seed3407` 先用于筛查趋势
- 只有在接近、敏感或准备下强结论时才补三次重复

### 4.6 第 4 周：投稿增强项

只有前面三周都过线，才考虑这一步。

当前增强项固定为：

1. `nnU-Net`
2. `TransUNet`
3. `CRAG` 上补 `1` 个最强外部方法
4. `LKMA kernel / Boundary width / lambda_dist` 的轻量敏感性分析

如果保底主实验都还没齐，不要做这一步。

## 6. 现在最容易犯的错

### 6.1 还没固定 split 就开始训练

这是最常见的问题。

后果是：

- 后面每次结果都可能不公平
- 你自己都说不清哪一版结果对应哪个数据划分

### 6.2 UNet 还没稳定就切主 baseline

这会导致你不知道问题到底来自：

- 数据链
- 训练协议
- 还是模型结构

### 6.3 baseline 还没稳定就开始堆模块

这会导致：

- 模块贡献不可解释
- 结果好坏无法归因
- 后面写论文会非常难讲

### 6.4 没有统一记录

如果你不统一 run 名称、配置文件和结果表，后面主表一定会乱。
