# 结直肠腺体分割主线实验矩阵与变更边界

这份文档不再只是列实验顺序，而是把主线实验写成真正的科研 protocol。

它要回答的不是“接下来跑什么”，而是下面八件事：

1. 每个阶段要验证的科学问题是什么
2. 这个阶段直接依据哪些论文
3. 这些论文具体支持什么实验设计
4. 哪些候选范围来自论文与工程约束，不是拍脑袋
5. 每个阶段唯一允许变化的变量是什么
6. 哪些固定项绝对不能一起变
7. 什么结果支持保留，什么结果只能降级或删除
8. 外部对比与 `CRAG` 补充验证怎样才算公平、成立、可写入正文

如果一轮实验回答不了上面八件事，就不算科学执行版实验。

---

## 1. 当前实验包的科学定位

### 1.1 这条主线要验证的不是“堆模块能不能涨点”

整个项目真正要验证的是一条连续证据链：

1. `UNet` 先证明数据链、训练链、评估链和可视化链是通的
2. `ResNet34-U-Net` 证明更强编码器是否值得作为正式比较基座
3. `LKMA` 验证更大有效感受野和更强形态建模是否有助于腺体整体轮廓表达
4. `Boundary Head` 验证显式边界监督是否有助于黏连腺体分离
5. `Distance-aware Loss` 验证边界附近的额外约束是否还能提供补充收益
6. 第一批外部对比验证你的方法是否优于有代表性的公开基线
7. `CRAG` 验证主结论在第二个同任务 benchmark 上是否还能成立

因此，主线不是“做五个模型”，而是：

- 先验证流程
- 再验证基座
- 再验证形态建模
- 再验证边界分离
- 最后再验证补充约束

### 1.2 当前总实验包分三层

第一层是保底可投稿核心实验：

1. `UNet` 单 seed 跑通
2. `UNet` 三次重复
3. `ResNet34-U-Net` 三次重复
4. `ResNet34-U-Net + LKMA`
5. `ResNet34-U-Net + LKMA + Boundary Head`
6. `ResNet34-U-Net + LKMA + Boundary Head + Distance-aware Loss` 的保留/删除结论
7. 第一批外部对比：`UNet++ / DeepLabV3+ / Attention U-Net`
8. `CRAG` 补充验证：`UNet / ResNet34-U-Net / 最终主方法`

第二层是投稿增强实验：

1. `nnU-Net`
2. `TransUNet`
3. `CRAG` 上补 `1` 个最强外部方法
4. `LKMA kernel / Boundary width / lambda_dist` 的轻量敏感性分析

第三层是当前不建议做的扩展：

- `Swin-Unet`
- foundation model / prompt / SAM
- diffusion
- 大规模 stain normalization
- 大规模后处理搜索
- 多数据集泛化式扩展

### 1.3 当前唯一正式顺序

当前只允许按下面顺序推进：

1. `UNet`
2. `ResNet34-U-Net`
3. `ResNet34-U-Net + LKMA`
4. `ResNet34-U-Net + LKMA + Boundary Head`
5. `ResNet34-U-Net + LKMA + Boundary Head + Distance-aware Loss`
6. 第一批外部对比
7. `CRAG` 补充验证
8. 投稿增强实验

这里之所以把第一批外部对比放在 `CRAG` 之前，是因为当前稿件主结论仍然以 `GlaS` 为中心，第一批外部对比属于主表核心证据，`CRAG` 属于第二 benchmark 补充验证。

## 2. 全阶段共享的固定协议

### 2.1 论文已经帮你锁定的固定项

下面这些不是临时经验值，而是由 `GlaS Challenge`、`MILD-Net`、`U-Net`、`UNet++`、`nnU-Net`、`DCAN` 等文献共同收缩出的默认执行口径：

- 主数据集：`GlaS`
- 第二 benchmark：`CRAG`
- 输入尺寸：`512 x 512`
- 图像：`RGB`
- 标签：二值语义 mask
- 主 loss：`BCE + Dice`
- optimizer：`AdamW`
- learning rate：`1e-3`
- scheduler：`ReduceLROnPlateau`
- epoch 上限：`120`
- early stopping patience：`20`
- AMP：开启
- threshold：先按 `0.5` 起步，再在验证集锁定
- repeat seeds：`3407 / 1234 / 2025`

这里还要进一步区分三类来源：

- `论文直接固定`
  - `GlaS` 作为主 benchmark、`CRAG` 作为第二 benchmark，以及 `F1 / Object Dice / Object Hausdorff` 作为主评价协议，都可直接落到执行层（`GlaS_Challenge_Contest_2017.pdf`, p.4, p.8-p.10；`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.9, p.12）
- `论文支持的候选范围`
  - `512 x 512` 输入、`BCE + Dice` 类损失组合、翻转/旋转/颜色扰动/弹性形变等增强，都有明确任务内或医学分割出处，但不是单一必须值（`UNet++_2018.pdf`, p.5；`SCAU-Net_Spatial-Channel_Attention_U-Net_for_Gland_Segmentation_2020.pdf`, p.5；`U-Net_2015.pdf`, p.6；`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.9）
- `工程默认起点`
  - `AdamW / 1e-3 / ReduceLROnPlateau / epoch 120 / early stopping 20 / AMP / seed3407` 这组值是为了统一首轮闭环、保证后续公平比较而定；当前提取论文只给出了相邻量级参考，而没有唯一固定值，因此它们必须在执行文档里被标注为工程协议，而不是伪装成论文原值

### 2.2 为什么当前先用这套默认训练协议

这套口径来自下面三条证据链，而不是为了省事：

1. `GlaS Challenge` 与 `MILD-Net` 已经明确该任务核心评价应围绕 `F1 / Object Dice / Object Hausdorff`，说明对象级和边界级质量比单纯像素 Dice 更关键（`GlaS_Challenge_Contest_2017.pdf`, p.8-p.10；`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.12）。
2. `UNet++` 与 gland-specific 的 `SCAU-Net` 都直接采用 `CE/BCE + Dice` 组合，因此第一版先用 `BCE + Dice` 合理（`UNet++_2018.pdf`, p.5；`SCAU-Net_Spatial-Channel_Attention_U-Net_for_Gland_Segmentation_2020.pdf`, p.5）。
3. 当前项目主目标是先建立统一、公平、可重复的协议，不是做训练技巧搜索，所以先固定简单稳定协议，再在模块层做单变量验证。

### 2.3 全阶段共享的不变项

除非某一阶段明确说明可以变，否则下面内容都不能变：

- split
- 输入尺寸
- 主 loss
- optimizer
- scheduler
- 主增强包
- threshold 锁定规则
- 是否使用 `TTA`
- 是否使用复杂后处理
- 评价指标集合

### 2.4 统一评估口径

统一评估规则固定为：

- `best checkpoint` 只按验证集选
- `GlaS TestA / TestB` 只做最终评估
- `CRAG test40` 只做最终评估
- 主指标：`F1 / Object Dice / Object Hausdorff`
- 辅助指标：`Dice / IoU / HD95 / Boundary F1`

这样做的依据来自：

- `GlaS Challenge`：对象级和边界级指标是主协议
- `Current Trends of AI for CRC Pathology Image Analysis`：后续腺体分割工作也围绕这组指标展开
- `DCAN / MILD-Net / TA-Net`：都表明黏连分离与边界质量不能只用像素指标判断

## 3. 阶段 A：`UNet` 单 seed 跑通

### 3.1 本阶段研究问题

当前要回答的问题不是“`UNet` 强不强”，而是：

- 数据读取链是否正确
- 训练和验证是否正常
- 指标脚本是否可信
- 预测图是否与指标一致

### 3.2 直接论文依据

- `U-Net`
- `GlaS Challenge`
- `Semantic Segmentation of Colon Glands with DCNN and Total Variation Segmentation`

这些论文支持的不是超参数精调，而是：

- `UNet` 作为腺体分割和医学分割的最小流程基线是合理的（`U-Net_2015.pdf`, p.1-p.2）
- `GlaS` 的任务与指标定义已经成熟（`GlaS_Challenge_Contest_2017.pdf`, p.4, p.8-p.10）
- 腺体任务必须先确认模型能分出基本 gland region，而不是一上来堆复杂结构

### 3.3 当前正式 run

- `A1_UNet_GlaS_v1_seed3407`

### 3.4 本阶段唯一允许变化

- 修数据 bug
- 修标签变换 bug
- 修指标脚本 bug

### 3.5 本阶段不允许变化

- 模型
- split
- 输入尺寸
- 主 loss
- 主增强包
- 测试集 threshold

### 3.6 本阶段必须产出

- `training_log.csv`
- `val_metrics.csv`
- `testA_metrics.csv`
- `testB_metrics.csv`
- `summary.md`
- `error_cases.md`
- 一组预测图

### 3.7 本阶段过线标准

- 训练、验证、测试都跑通
- 没有大面积全背景或全白预测
- 没有明显系统性错位
- 指标和图不矛盾
- 结果完整落盘

### 3.8 不过线时的科学解释

如果这一步不过线，不能得出“`UNet` 不适合这个任务”，只能说明：

- 数据协议可能有问题
- 标签处理可能有问题
- 评估脚本可能有问题
- 当前实验闭环尚未建立

## 4. 阶段 B：`UNet` 三次重复

### 4.1 本阶段研究问题

要回答的是：

- 当前统一协议是否稳定
- `UNet` 作为最小流程基线能否复现
- 当前波动范围大致在哪个区间

### 4.2 直接论文依据

- `U-Net`
- `nnU-Net`
- `Current Trends of Artificial Intelligence for Colorectal Cancer Pathology Image Analysis`

这些论文共同支持一个判断：

- 在医学分割里，性能差异常常不只来自结构，也来自 pipeline 稳定性（`nnU-Net_a_self-configuring_method_for_deep_learning-based_biomedical_image_segmentation_2021.pdf`, p.3-p.5）
- 所以在加模块前先得到可复现的最小基线，是科学实验而不是浪费时间（`Current_Trends_of_Artificial_Intelligence_for_Colorectal_Cancer_Pathology_Image_Analysis_A_Systematic_Review_2020.pdf`, p.4, p.6-p.7）

### 4.3 当前正式 run

- `B1_UNet_GlaS_v1_seed3407`
- `B2_UNet_GlaS_v1_seed1234`
- `B3_UNet_GlaS_v1_seed2025`

### 4.4 本阶段唯一允许变化

- 只允许修正已确认 bug

### 4.5 本阶段不允许变化

- split
- augmentation
- batch size
- threshold
- 选模指标

### 4.6 本阶段必须产出

- `unet_mean_std_table.csv`
- `unet_failure_cases.md`
- `unet_stage_summary.md`

### 4.7 本阶段过线标准

- 三次都跑通
- 已形成 `mean +- std`
- 波动不过大
- 失败模式大致一致

### 4.8 不过线时的科学解释

如果三次波动失控，不允许把问题归因到“数据集太小”，必须优先检查：

- 协议是否漂移
- 随机增强是否太强
- 数据加载或 checkpoint 选择是否不一致

## 5. 阶段 C：`ResNet34-U-Net` 三次重复

### 5.1 本阶段研究问题

要回答的是：

- 更强残差编码器是否值得成为全文主 baseline
- 结构增强本身是否带来比 `UNet` 更稳定的结果

### 5.2 直接论文依据

- `Deep Residual Learning for Image Recognition`
- `U-Net`
- `UNet++`
- `Attention U-Net`
- `DeepLabV3+`

这些论文支持的不是“`ResNet34-U-Net` 一定最强”，而是：

- 用残差编码器替换普通编码器是有标准出处的（`Deep_Residual_Learning_for_Image_Recognition_2016.pdf`, p.2-p.5）
- 在你的主方法前，必须先有一个比 `UNet` 更正式的比较基座
- 后续外部对比和模块消融都需要围绕它展开

### 5.3 当前正式 run

- `C1_R34UNet_GlaS_v1_seed3407`
- `C2_R34UNet_GlaS_v1_seed1234`
- `C3_R34UNet_GlaS_v1_seed2025`

### 5.4 本阶段唯一允许变化

- 从 `UNet` 切换到 `ResNet34-U-Net`

### 5.5 本阶段不允许变化

- split
- 输入尺寸
- augmentation
- loss
- threshold
- seed 规则

### 5.6 本阶段必须产出

- `r34unet_mean_std_table.csv`
- `unet_vs_r34unet_compare.csv`
- `r34unet_failure_cases.md`

### 5.7 本阶段过线标准

- 平均结果不低于 `UNet`
- 三次方向一致
- 可作为主 baseline

### 5.8 不过线时的科学解释

如果 `ResNet34-U-Net` 不稳定，不说明路线错了，只说明：

- backbone 接入实现可能有问题
- 预训练策略可能不统一
- 当前协议尚不足以支撑更强主基座

这时不准进入 `LKMA`。

## 6. 阶段 D：`+ LKMA`

### 6.1 本阶段研究问题

要回答的是：

- 更大的有效感受野和更强形态建模，是否能改善腺体整体轮廓表达
- 这种改善是否主要体现在对象级结果，而不是只体现在像素重叠

### 6.2 直接论文依据

- `Large Kernel Matters`
- `Scaling Up Your Kernels to 31x31`
- `Wavelet Convolutions for Large Receptive Fields`
- `Morphology-Inspired Unsupervised Gland Segmentation via Selective Semantic Grouping`
- `MILD-Net`

这些论文共同支持：

- 腺体任务存在形态变化大、上下文范围需求强的问题（`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.3, p.17）
- 大核与更大有效感受野是合理改进方向（`Large_Kernel_Matters_GCN_2017.pdf`, p.1, p.3-p.5）
- 但第一轮不应把大核实验做成位置和结构的无上限搜索

### 6.3 当前候选范围为什么是这样

当前只允许：

- 默认位置：`bottleneck`
- 备选位置：最多再补 `1` 个相邻高层位置
- kernel 候选：`15 / 21`

原因：

- `Large Kernel Matters` 已证明 `k=15` 是最明确的有效大核起点（`Large_Kernel_Matters_GCN_2017.pdf`, p.5）
- 更大感受野路线在后续大核文献中持续被支持，但在当前提取索引里，最明确的数值锚点仍然是 `k=15`；因此 `21` 只作为一次补充候选，不扩成完整 kernel search
- 你的目标是验证“形态建模是否成立”，不是做 kernel search 论文

### 6.4 当前正式 run

- `D1_R34UNet_LKMA_GlaS_posB_k15_seed3407`
- `D2_R34UNet_LKMA_GlaS_posB_k15_seed1234`
- `D3_R34UNet_LKMA_GlaS_posB_k15_seed2025`

只有首轮趋势明确时，才允许补：

- `D4_R34UNet_LKMA_GlaS_posB_k21_seed3407`

### 6.5 本阶段唯一允许变化

- `LKMA` 插入
- 在受限候选内试 `kernel`

### 6.6 本阶段不允许变化

- backbone
- loss
- augmentation
- threshold
- split
- `Boundary Head`
- `Distance-aware Loss`

### 6.7 本阶段必须产出

- `lkma_compare_table.csv`
- `lkma_cost_record.csv`
- `lkma_visual_compare/`

### 6.8 本阶段保留标准

- `Object Dice` 或 `F1` 稳定提升
- 多个 seed 方向一致
- 边界指标不明显退化
- 可视化中腺体整体轮廓更完整

### 6.9 本阶段删除标准

- 提升接近波动范围
- 成本明显增加但效果不稳定
- 只能在单次 seed 上变好

### 6.10 不过线时的科学解释

如果 `LKMA` 不成立，不代表形态信息不重要，更可能代表：

- 当前实现位置不合适
- 大核收益在当前基座上不稳定
- 形态建模的增益不足以覆盖成本

这时允许删除 `LKMA`，但不允许继续扩更多大核变体。

## 7. 阶段 E：`+ Boundary Head`

### 7.1 本阶段研究问题

要回答的是：

- 显式边界监督是否真能改善黏连腺体分离
- 这种改善是否能在边界指标、对象级指标或局部可视化中体现

### 7.2 直接论文依据

- `DCAN`
- `Gated-SCNN`
- `TA-Net`
- `Gland Segmentation via Dual Encoders and Boundary-Enhanced Attention`
- `MILD-Net`

这些论文共同支持：

- 腺体任务的关键难点之一是边界模糊和黏连分离（`DCAN_Deep_Contour-Aware_Networks_for_Accurate_Gland_Segmentation_2016.pdf`, p.1-p.5；`TA-Net_Topology-Aware_Network_for_Gland_Segmentation_2022.pdf`, p.1-p.3）
- 显式边界分支是有先例的，不是随便加个支路（`Gated-SCNN_Gated_Shape_CNNs_for_Semantic_Segmentation_2019.pdf`, p.1-p.4；`Gland_Segmentation_via_Dual_Encoders_and_Boundary-Enhanced_Attention_2024.pdf`, p.2-p.4）
- 首轮应使用轻量单边界分支，而不是一开始就做复杂拓扑和多分支设计

### 7.3 当前候选范围为什么是这样

首轮只允许：

- 单边界分支
- 边界宽度：`3`
- 备选边界宽度：`5`
- 边界 loss：`BCE`
- 边界 loss 权重：首轮 `0.3`
- 备选权重：`0.2 / 0.3 / 0.5`

原因：

- `DCAN` 和 `Gated-SCNN` 证明边界/轮廓单独建模有效（`DCAN_Deep_Contour-Aware_Networks_for_Accurate_Gland_Segmentation_2016.pdf`, p.1-p.5；`Gated-SCNN_Gated_Shape_CNNs_for_Semantic_Segmentation_2019.pdf`, p.1-p.4）
- 当前项目目标是验证边界监督是否成立，而不是比较各种边界标签生成法
- 因此候选范围必须小而可解释
- `w=3/5` 与 `loss weight=0.2/0.3/0.5` 当前都被明确标成`工程候选范围`，它们是为了把搜索空间收窄到一次可解释试验，而不是声称已有单篇腺体论文把这几个数字完全锁死

### 7.4 当前正式 run

- `E1_R34UNet_LKMA_Boundary_GlaS_w3_l03_seed3407`
- `E2_R34UNet_LKMA_Boundary_GlaS_w3_l03_seed1234`
- `E3_R34UNet_LKMA_Boundary_GlaS_w3_l03_seed2025`

如首轮有趋势，最多补：

- `E4_R34UNet_LKMA_Boundary_GlaS_w5_l03_seed3407`
- `E5_R34UNet_LKMA_Boundary_GlaS_w3_l05_seed3407`

### 7.5 本阶段唯一允许变化

- 是否加入 `Boundary Head`
- 边界宽度试一次
- 边界权重试一次

### 7.6 本阶段不允许变化

- `LKMA` 结构
- 主 loss
- augmentation
- split
- threshold
- `Distance-aware Loss`

### 7.7 本阶段必须产出

- `boundary_main_and_edge_table.csv`
- `boundary_local_zoom/`
- `adhesion_case_compare/`

### 7.8 本阶段保留标准

至少满足下面之一：

- `Boundary F1` 提升
- `HD95` 或 `Object Hausdorff` 改善
- 黏连样本分离更好

并同时满足：

- 主指标不明显下降
- 可视化支持结论

### 7.9 本阶段删除标准

- 边界指标不改善
- 可视化不支持
- 只是轮廓更碎，没有真正分离更好

### 7.10 不过线时的科学解释

如果边界分支不成立，不说明边界问题不存在，而更可能说明：

- 当前边界标签定义不合适
- 边界监督强度过弱或过强
- 当前主干已经能覆盖大部分边界收益

这时不建议直接叠 `Distance-aware Loss`。

## 8. 阶段 F：`+ Distance-aware Loss`

### 8.1 本阶段研究问题

要回答的是：

- 边界附近的额外距离约束是否还能带来补充收益
- 它的收益是否足够稳定，值得进入最终主模型

### 8.2 直接论文依据

- `Distance Map Loss Penalty Term for Semantic Segmentation`
- `Efficient Segmentation Learning Downsampling Near Semantic Boundaries`

这些论文支持：

- 边界附近的错误可以被额外惩罚（`Distance_Map_Loss_Penalty_Term_for_Semantic_Segmentation_2019.pdf`, p.1-p.5）
- 距离图监督更像补充约束，不应先于主结构问题被解决

### 8.3 当前候选范围为什么是这样

首轮只允许：

- 距离图来自二值 gland mask 的 Euclidean distance map
- 归一化到 `[0, 1]`
- `lambda_dist`：`0.05 / 0.1 / 0.2`

原因：

- `Distance Map Loss Penalty Term` 直接使用由真值 mask 生成的 distance map，并把它作为边界惩罚项（p.1-p.3）
- 这两篇距离约束文献都强调边界附近惩罚，而不是复杂方向场建模
- 当前项目目标是判断“值不值得保留”，不是再开一个距离图方法子课题
- 因此 `Euclidean distance map + [0,1] 归一化 + lambda 0.05/0.1/0.2` 在当前执行层里被标记为`工程候选范围`，不是论文直接固定值

### 8.4 当前正式 run

- `F1_R34UNet_LKMA_Boundary_Dist_GlaS_l005_seed3407`
- `F2_R34UNet_LKMA_Boundary_Dist_GlaS_l01_seed3407`
- `F3_R34UNet_LKMA_Boundary_Dist_GlaS_l02_seed3407`

只有首轮有潜力时，才补：

- `F4_R34UNet_LKMA_Boundary_Dist_GlaS_best_seed1234`
- `F5_R34UNet_LKMA_Boundary_Dist_GlaS_best_seed2025`

### 8.5 本阶段唯一允许变化

- `lambda_dist`

### 8.6 本阶段不允许变化

- 边界分支配置
- `LKMA` 配置
- 主 loss
- split
- threshold

### 8.7 本阶段必须产出

- `distance_compare_table.csv`
- `distance_failure_cases.md`
- `distance_keep_or_drop.md`

### 8.8 本阶段保留标准

- 至少一个对象级或边界级主指标稳定提升
- 提升不只是单次波动
- 失败案例没有明显变多

### 8.9 本阶段降级标准

- 收益接近波动范围
- 稳定性变差
- 可视化看不出额外帮助

### 8.10 当前默认态度

- 它是可选增强项
- 不是默认主贡献

## 9. 第一批外部对比固定方案

### 9.1 本阶段研究问题

要回答的是：

- 你的最终方法是否优于代表性公开基线
- 这种优势是否来自公平协议，而不是额外技巧

### 9.2 直接论文依据

- `UNet++`
- `DeepLabV3+`
- `Attention U-Net`
- `nnU-Net`
- `TransUNet`

这些论文的作用分别是：

- `UNet++`：更强 U 形跳连代表（`UNet++_2018.pdf`, p.1-p.5）
- `DeepLabV3+`：强通用 encoder-decoder 代表
- `Attention U-Net`：轻量注意力增强代表（`Attention_U-Net_2018.pdf`, p.1-p.7）
- `nnU-Net / TransUNet`：投稿增强对比来源

### 9.3 第一批固定名单

这一批是保底必须完成的：

1. `UNet++`
2. `DeepLabV3+`
3. `Attention U-Net`

### 9.4 第二批增强名单

这一批只补 `1-2` 个：

1. `nnU-Net`
2. `TransUNet`

`Swin-Unet` 当前不进入第一版执行包。

### 9.5 公平比较协议

外部对比必须统一：

- 同一 `GlaS` split：`train68 / val17 / testA60 / testB20`
- 同一输入尺寸：`512 x 512`
- 同一训练轮次上限：`120`
- 同一 early stopping patience：`20`
- 同一基础增强包
- 同一 threshold 规则：在 `val17` 定，再固定到 `TestA / TestB`
- 同一评估指标：`F1 / Object Dice / Object Hausdorff / Dice / IoU / HD95 / Boundary F1`

此外必须遵守：

- 优先使用公开实现或作者实现
- 所有适配改动写入 `external_reproduction_note.md`
- 不允许为了迁就某个外部方法单独改 split、单独加后处理、单独开 `TTA`

### 9.6 外部对比跑几次

最低要求：

- 第一批每个方法至少跑 `seed3407`

必须补三次重复的条件：

- 最终方法相对该方法在任一主指标上的差距 `<= 1.5` 个百分点
- 两者在可视化上差异不明显
- 该方法是主表第二强方法

补跑后形式：

- `seed3407 / 1234 / 2025`
- 重新汇报 `mean +- std`

### 9.7 本阶段必须产出

- `external_protocol_table.csv`
- `external_main_result_table.csv`
- `external_reproduction_note.md`

### 9.8 本阶段通过线

- 第一批三种方法全部完成
- 使用统一 split 与统一评估口径
- 至少能解释各方法的强弱和差异风格
- 结果不是只挑对自己有利的一版

### 9.9 本阶段止损规则

如果某个外部方法复现成本明显失控：

- 先查数据接口和训练协议是否统一
- 如仍无法控制成本，则降级为“引用对比”
- 但必须明写原因，不允许直接换成别的方法假装没发生

## 10. `CRAG` 补充验证固定方案

### 10.1 本阶段研究问题

要回答的是：

- 在第二个同任务 benchmark 上，主线结论是否还能基本成立
- 最终模型相对 baseline 的优势是否没有完全消失

### 10.2 直接论文依据

- `MILD-Net`
- `GlaS Challenge`
- `Current Trends of Artificial Intelligence for Colorectal Cancer Pathology Image Analysis`

这些论文支持：

- `CRAG` 是腺体任务的第二 benchmark，而不是随便找来的补充数据（`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.2, p.9）
- `GlaS + CRAG` 作为双数据集证据链是有文献传统的（`MILD-Net_Minimal_Information_Loss_Dilated_Network_2018.pdf`, p.12；`Current_Trends_of_Artificial_Intelligence_for_Colorectal_Cancer_Pathology_Image_Analysis_A_Systematic_Review_2020.pdf`, p.4）
- `CRAG` 更适合写成同任务补充验证，而不是强泛化

### 10.3 当前正式协议

固定读取：

- `crag_train153.csv`
- `crag_val20.csv`
- `crag_test40.csv`

### 10.4 最低必须跑的模型

1. `UNet`
2. `ResNet34-U-Net`
3. 最终主方法

### 10.5 增强验证可选项

如果资源允许，再补：

1. 第一批外部对比里在 `GlaS` 上最强的 `1` 个方法
2. `nnU-Net` 或 `TransUNet` 中你实际跑过的最强增强方法 `1` 个

### 10.6 公平比较协议

`CRAG` 阶段必须统一：

- 同一 split：`train153 / val20 / test40`
- 同一输入尺寸：`512 x 512`
- 同一主 loss：`BCE + Dice`
- 同一 optimizer：`AdamW`
- 同一 scheduler：`ReduceLROnPlateau`
- 同一 threshold 规则：先在 `val20` 定，再固定用于 `test40`
- 同一评估指标：`F1 / Object Dice / Object Hausdorff / Dice / IoU / HD95 / Boundary F1`

### 10.7 `CRAG` 跑几次

当前最低要求：

- `UNet`：`seed3407`
- `ResNet34-U-Net`：`seed3407`
- 最终主方法：`seed3407`

必须补三次重复的条件：

- 主方法比 baseline 优势 `<= 1.5` 个百分点
- 主方法只在像素指标上好，但对象级指标不明显
- 你准备把 `CRAG` 写进正文主表

补跑后形式：

- `seed3407 / 1234 / 2025`

### 10.8 本阶段必须产出

- `crag_main_result_table.csv`
- `glas_crag_joint_table.csv`
- `crag_success_cases/`
- `crag_failure_cases/`
- `crag_stage_summary.md`

### 10.9 本阶段通过线

- 最终模型在 `CRAG` 上没有崩溃式下降
- 相对 baseline 的优势没有完全消失
- 至少 `2` 个主/辅指标方向与 `GlaS` 保持一致
- 失败类型可解释

### 10.10 本阶段不过线怎么办

如果下面任意成立，就不能把 `CRAG` 写成强支撑证据：

- 比 baseline 明显更差
- 趋势完全反向
- 结果极不稳定
- 解释不清为什么崩

处理方式固定为：

- 稿件叙事降级为“GlaS 主结论 + CRAG 补充验证”
- 不再继续堆模块
- 不写“强泛化”

## 11. 变量变更边界

### 11.1 什么叫允许改

只有下面两类才叫允许改：

- 当前阶段唯一新增的变量
- 已经确认是 bug 的实现问题

### 11.2 什么叫不允许改

下面这些如果一起变了，就不再是公平对比：

- split 变了
- 输入尺寸变了
- batch size 变了
- augmentation 变了
- loss 变了
- threshold 变了
- 是否使用后处理变了
- 是否使用 `TTA` 变了

### 11.3 如果确实要改怎么办

必须同时满足：

1. 在实验记录中写清原因
2. 从新的版本重新起 baseline
3. 不和旧协议主表混在一起

## 13. 什么时候说明你已经有一套可投稿主实验

至少满足下面这些：

- `ResNet34-U-Net` 稳定
