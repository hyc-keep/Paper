# 论文实验项目文档编写规范(完整版)

> 这份文件只负责模板层总规范,不再负责总路由。

## 正式角色

本文件只负责 3 件事:

1. 规定项目内模板层怎么分工
2. 规定模板层的状态口径和门禁口径
3. 规定正式模板该怎么互相衔接

它不负责决定“现在该进研究、编码还是文档阶段”。
那件事只归根目录阶段 skill 管。

## 模板层固定分工

### A. 正式模板

- `模板R_研究定标记录.md`: 锁定当前阶段为什么现在研究、读了什么、提取了什么硬约束
- `模板0_阶段实现卡.md`: 锁定当前阶段为什么现在开始、允许改什么、禁止改什么
- `模板1_阶段总协议.md`: 锁定阶段目标、边界、通过标准、回退条件
- `模板2_设计依据.md`: 锁定设计取舍和文献依据
- `模板3_阶段验收.md`: 负责在 `3A` 和 `3B` 之间选型
- `模板3A_训练评估型阶段验收.md`: 用于会产生 run、metrics、可视化证据的阶段
- `模板3B_治理汇总型阶段验收.md`: 用于汇总、裁决、止损、交接类阶段
- `模板4_文献证据提取.md`: 提取论文和参考代码证据

### B. Guard 留痕

- `模板5_Pre-check提取.md`
- `模板6_阶段门控检查.md`
- `模板7_当前代码库状态.md`
- `模板8_Pre-check Guard.md`
- `模板9_diagnostics_result.txt模板.md`
- `模板10_Post-QC Guard.md`

这些文件只负责门禁、留痕和质检,不负责替代正式协议或学习型说明文。

### C. 学习型说明文规程

- `实验执行文档编写规范.md`
- `TCGA原始标杆对齐清单.md`
- `学习型说明文人工审稿清单.md`

这些文件共同负责 `implementation_tracking` 学习型说明文的自动门禁、原始标杆对照和人工审稿。

## 三套状态口径

- `Stage Gate`: `allow / blocked`
- `Diagnostics / Post-QC`: `pass / partial / fail / not_applicable`
- `implementation_tracking` 阶段入口: `not_started / blocked / partial / pass`

禁止把这三套状态混成一个总状态。

## 三类硬门禁

### 1. Pre-check 门禁

只要本轮已经形成 `Pre-check` 四件套,就必须运行:

`python b_class_auxiliary/tools/check_precheck_docs.py --precheck-guard [Pre-check Guard相对路径]`

### 0. 研究定标门禁

只要本轮准备把研究阶段口头写成“已完成”,就必须先运行:

工具命令: `b_class_auxiliary/tools/check_research_alignment_gate.py`

只有当研究门禁报告 `research_alignment_gate_report.md` 为 `pass`,后面的阶段锁定才允许被视为正式开始。

### 0.5 阶段锁定门禁

只要本轮准备进入正式编码、正式配置修改或正式资产修改,就必须先运行:

工具命令: `b_class_auxiliary/tools/check_stage_definition_gate.py`

只有当阶段锁定门禁报告 stage_definition_gate_report.md 为 `pass`,后面的 `Pre-check` 才允许被视为正式通过。

### 2. 代码质量门禁

只要本轮修改了正式代码、配置或正式资产,就必须运行:

`python b_class_auxiliary/tools/check_code_quality_gate.py --post-qc-guard [Post-QC Guard相对路径]`

最小物理证据至少包含:

- smoke run
- dataloader batch
- tensor shape / dtype
- loss finite
- backward / optimizer.step

### 3. 文档门禁

- 更新学习型说明文后:  
  `python b_class_auxiliary/tools/check_learning_docs.py --precheck-guard [Pre-check Guard相对路径] --post-qc-guard [Post-QC Guard相对路径]`
- 修改正式模板、规程或 `01_实验执行` 正式协议后:  
  `python b_class_auxiliary/tools/check_formal_stage_docs.py --post-qc-guard [Post-QC Guard相对路径]`

## 模板使用顺序

### 研究定标

如果当前阶段还没完成正式研究定标:

1. 先用 `模板R_研究定标记录.md` 起草正式输出名 `研究定标记录.md`
2. 运行 `b_class_auxiliary/tools/check_research_alignment_gate.py` 拿到 `research_alignment_gate_report.md`
3. 研究结论只能回答 `allow_stage_lock / blocked`
4. 只有当研究门禁报告已经裁成 `pass` 时,才允许开始阶段锁定

### 阶段锁定

如果当前阶段正式协议还没建立:

1. 先用 `模板0_阶段实现卡.md` 起草阶段卡正式输出名 00_阶段实现卡.md
2. 运行 `b_class_auxiliary/tools/check_stage_definition_gate.py` 拿到阶段锁定门禁报告 stage_definition_gate_report.md
3. 用 `模板1_阶段总协议.md` 起草正式阶段协议
4. 用 `模板2_设计依据.md` 和 `模板4_文献证据提取.md` 锁定依据

### 编码前

1. 先确认阶段锁定门禁报告 stage_definition_gate_report.md 为 `pass`
2. 再补 `模板5` 到 `模板8`
3. 通过 Pre-check 后才允许进入正式代码

### 编码后

1. 补 `模板9`
2. 补 `模板10`
3. 跑代码质量门禁
4. 需要时再更新学习型说明文

## 冲突裁决

如果模板层内部说法冲突,按下面顺序裁决:

1. `00_技能体系导航.md`
2. 本文件
3. `模板1` 到 `模板10`
4. `实验执行文档编写规范.md`

## 当前边界

- 本文件不替代根目录 `crc-gland-coding-guard`
- 本文件不负责直接判某个具体任务是否已经 `pass / partial / fail / not_applicable`
- 本文件只负责模板层分工、Guard 留痕和学习型说明文规程之间的接口关系
- 本文件必须把代码落点、工程落点、验收/回退与自动门禁一起说清
- 典型工程落点包括 `b_class_auxiliary/coding_guards/[任务目录]/`、`reports/stage_reports/implementation_tracking/[阶段]/`、`crc_gland_segmentation_project/tools/`

## 八、强制执行机制

- 研究定标先产出 `研究定标记录.md`,再进入阶段锁定
- 阶段锁定通过后再进入 `Pre-check`
- `b_class_auxiliary/tools/check_research_alignment_gate.py`、`b_class_auxiliary/tools/check_stage_definition_gate.py`、`b_class_auxiliary/tools/check_precheck_docs.py`、`b_class_auxiliary/tools/check_learning_docs.py`、`b_class_auxiliary/tools/check_formal_stage_docs.py` 必须在本文件中保持正式命令可见
- `模板R_研究定标记录.md`、`模板0_阶段实现卡.md`、`TCGA原始标杆对齐清单.md`、`学习型说明文人工审稿清单.md` 必须继续保留在总规范视野内

## 红线

- 不允许把本文件写成新的总入口
- 不允许拿模板层给研究或编码提前放行
- 不允许跳过 `研究定标记录.md` 直接进入阶段锁定
- 不允许跳过阶段卡正式输出名 00_阶段实现卡.md 和阶段锁定门禁报告 stage_definition_gate_report.md 直接进入 Pre-check
- 不允许让 Guard 留痕替代真实代码证据
- 不允许让学习型说明文替代正式协议或验收
