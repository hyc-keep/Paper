# 模板5: Pre-check提取

> **使用场景**: 每次正式编码或正式模板修订前,先把规划约束、路线约束和参考依据提成结构化表格。
> **调用顺序**: 先看 `00_技能体系导航.md`,再结合中央 `crc-gland-coding-guard` 的 Pre-check 要求使用本模板
> **与中央门禁的关系**: 本模板负责把“读到了什么、提取出了什么约束”结构化落盘; 它不替代 `stage_gate_check.md`、`current_codebase_状态.md` 和最终的 `pre_check_guard.md`

---

这份模板最容易被错误使用成“抄几条文件名就算前检完成”。

不对。

它真正要回答的是:
- 当前任务到底受哪些上游文件约束
- 这些约束里哪些是 `官方协议固定项 / 路线层已锁定 / 论文支持的候选范围 / 工程冻结规则`
- 当前任务唯一允许改动的变量是什么
- 如果现在越界,会破坏哪一层正式 protocol

```markdown
# Pre-check Extraction

## 1. 本次任务
- 任务名:
- 当前阶段:
- 上一阶段:
- 日期:

## 2. 规划约束提取

| 来源文件 | 章节 | 约束内容 | 约束类型 | 对本次实现的影响 |
|---------|------|---------|---------|----------------|
| `00_执行导航.md` | `§X` | [约束] | `官方协议固定项 / 路线层已锁定 / 论文支持的候选范围 / 工程冻结规则` | [影响] |
| `03_命名与结果记录规范.md` | `§X` | [约束] | `工程冻结规则` | [影响] |
| `04_评估口径与官方脚本对齐.md` | `§X` | [约束] | `官方协议固定项` | [影响] |
| `07_实验执行全局修订与质检规范.md` | `§X` | [约束] | `工程冻结规则` | [影响] |

**最小要求:**
- 这里不要只抄文件名,必须写“这一条约束具体限制了本轮什么”
- 如果当前任务是文档/模板治理,也必须明确写出为什么它仍然受命名规则、评估规则或质检规则约束
- 如果某个文件与你本轮任务确实无关,不要假装有关,而是写清“不适用”的原因

## 2.1 预期工程落点

| 对象层 | 预期落点 | 为什么现在就要写 |
|------|---------|----------------|
| guard 提取留痕 | `b_class_auxiliary/coding_guards/[YYYYMMDD_任务简述]/pre_check_extraction.md` | 防止前检做完却没有正式落盘 |
| 正式代码 | `src/[模块]/[文件].py` / `scripts/[脚本].py` / `tools/[脚本].py` | 让后面的 `Stage Gate` 和 `Pre-check Guard` 能核对本轮是不是越界到错误目录 |
| 配置与资产 | `configs/[配置].yaml` / `reports/[子目录]/[文件]` | 提前判断这轮是否会影响冻结字段、结果记录或质检链 |

**怎么写才算合格:**
- 这里写的是“本轮约束最终会落到哪些工程对象”,不是随手猜一个文件名
- 如果本轮只是修正式模板,也要写清楚工程落点是 `.trae/skills/[模板文件].md` 与 `b_class_auxiliary/coding_guards/[任务目录]/`
- 如果本轮不会碰代码,要明确写 `代码落点: not_applicable`,并解释为什么只影响模板或协议层

## 3. 路线层约束提取

| 来源文件 | 章节 | 当前结论 | 不允许做什么 |
|---------|------|---------|-------------|

**推荐写法:**
- `当前结论` 不要写成空话,例如“要严谨”
- 更稳的写法是: “当前阶段只能改 guard 模板层,不能顺手改训练协议本体”

## 4. 文献/参考实现提取

| 类型 | 来源 | 章节/文件/行号 | 可直接采用内容 | 本项目调整 |
|------|------|---------------|---------------|-----------|

**怎么理解这一节:**
- 如果本轮是代码实现,这里优先写论文、仓库、官方脚本
- 如果本轮是模板/文档治理,这里允许引用项目内导航、上一轮真实 guard 样例和中央门禁 skill
- 如果某类来源本轮确实不存在,允许写 `not_applicable`,但必须写明为什么不适用

## 5. 当前阶段唯一允许改动的变量
- 允许改:
- 不允许改:
- 如果越界会影响:

**红线提醒:**
- 不能把“本轮要改哪些文件”写成“当前阶段唯一允许改动的变量”
- 真正要写的是: 本轮只允许动哪一层 protocol,不能碰哪一层 protocol

## 5.1 本轮最小验证计划

| 验证项 | 计划动作 | 预期证据 |
|------|---------|---------|
| `py_compile / import` | [准备怎么检查] | [日志或命令输出路径] |
| `smoke run` | [最小运行命令] | [日志路径 + 成功信号] |
| `dataloader batch` | [准备抽查哪个 batch] | [shape / dtype / unique 值或数值范围] |
| `loss / backward` | [准备如何验证 finite 和单步优化] | [loss 数值 + backward / optimizer.step 证据] |

**怎么写才算合格:**
- 如果本轮存在正式代码改动,这里不能空着
- 这里写的是“本轮打算如何证明代码没写歪”,不是写一句“后面再检查”
- 如果本轮只是模板治理或纯说明文修订,允许写 `not_applicable`,但必须说明为什么本轮不存在正式代码改动
```
