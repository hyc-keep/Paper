# 模板3: 阶段验收总入口(阶段验收.md)

> **使用场景**: 每个阶段最后的正式放行文档入口,负责先判断“当前阶段属于哪种验收类型”,再选择对应子模板填写
> **调用顺序**: 先看 `00_技能体系导航.md`,并确保 `模板1~2` 与 `模板5~10` 已完成前置冻结和留痕
> **与中央门禁的关系**: 本文件只负责“选对阶段验收模板”; 真正的通过标准、物理证据和放行结论,仍要落到对应子模板正文里

---

## 单文档双达标模式（按需启用）

默认情况下,本文件只是“阶段验收该选 `3A` 还是 `3B`”的入口,不是教学长文本体。

只有当用户明确要求“阶段验收正文本身也要承担教学解释”时,才进入**单文档双达标模式**:

- 先继续用本文件完成 `3A / 3B` 选型
- 再到对应子模板中启用它们自己的“双达标附加块”
- 不允许停留在本入口模板里空谈“要讲清楚”,却不把增强块真正落到 `阶段验收.md`

**最小理解:**
- `模板3` 负责“选对赛道”
- `模板3A / 3B` 负责“把正文写对”
- 如果用户明确要“小白也能直接读懂阶段验收”,真正要补强的是 `3A / 3B` 的正文模板,不是只改本入口

---

## 1. 为什么要拆成两个子模板

之前把所有阶段都塞进同一份阶段验收模板里，最大的问题不是“写得不够多”，而是**任务类型混杂**：

- 训练/评估型阶段关心的是: `run`、`metrics`、多 seed 稳定性、可视化、对上游 baseline 的增益
- 治理/汇总型阶段关心的是: 结果表是否对齐、文档是否闭环、阶段结论是否真实、下游是否还能继续推进

如果继续共用一份大模板，就会出现两种坏事：
- 训练阶段被治理类条目打断，正文变得松散
- 汇总阶段被训练类字段污染，被迫硬填 `train_log.csv`、`testA_metrics.csv` 这类根本不适用的内容

所以现在改成：
- `模板3A_训练评估型阶段验收.md`
- `模板3B_治理汇总型阶段验收.md`

本文件以后不再承担全部正文，而是承担**模板选型入口**。

---

## 2. 先判断当前阶段属于哪一类

### 2.1 训练评估型阶段

满足下面任一特征，优先使用 `模板3A_训练评估型阶段验收.md`:

- 当前阶段会新增正式 `run`
- 当前阶段会产生新的 `val/test metrics`
- 当前阶段要比较 baseline / 模块增益 / 多 seed 稳定性
- 当前阶段要提交新的预测图、可视化、误差分析或测试结果

**典型阶段:**
- `02_UNet流程验证`
- `03_UNet稳定性`
- `04_Baseline`
- `05_LKMA`
- `06_Boundary`
- `07_Distance`
- `08_外部对比`
- `09_CRAG验证`

### 2.2 治理汇总型阶段

满足下面任一特征，优先使用 `模板3B_治理汇总型阶段验收.md`:

- 当前阶段主要任务不是再跑一个新模型，而是整理、归档、汇总、裁决
- 当前阶段更关心“已有结果是否够用、够真、够统一”
- 当前阶段主要交付物是表格、结论文档、止损判断、最终阅读入口，而不是新的 `run`

**典型阶段:**
- `01_数据协议` 的最终放行收口
- `10_结果汇总`
- `11_总验收与止损`

> 注意:
> `01_数据协议` 介于两者之间。
> 如果当前任务重点是“数据资产本身是否通过”，偏向治理汇总型；
> 如果重点是“数据资产是否已具备被训练阶段消费的正式资格”，也可以吸收 `模板3A` 的资产验证块。

## 4. 两类子模板共同必须满足的底线

无论使用 `3A` 还是 `3B`，下面这些底线都不能少：

- 最终结论只能写 `pass / partial / fail / not_applicable`
- 每个结论都必须给出物理证据（路径、字段、数值、图像或记录位置）
- 必须诚实写清楚“已完成 / 未完成 / 不适用”的边界
- 必须说明下游阶段是否还能继续，以及为什么
- 必须和 `Pre-check Guard`、`Post-QC Guard`、`implementation_tracking` 的说法保持一致

---

## 5. 正式使用方式

后续真正填写某个阶段的 `阶段验收.md` 时，流程固定为：

1. 先看本文件，确认当前阶段属于 `3A` 还是 `3B`
2. 再打开对应子模板
3. 按子模板填写正式 `阶段验收.md`
4. 回到 `Post-QC Guard`、阶段 `README.md` 和 `implementation_status.md` 做状态同步：
  - `Post-QC Guard` / `阶段验收.md` 继续使用 `pass / partial / fail / not_applicable`
   - `implementation_tracking` 阶段入口（`README.md` / `implementation_status.md`）只能使用 `not_started / blocked / partial / pass`
   - 不允许把局部 Diagnostics 结论直接抄成阶段入口状态

## 5.1 阶段验收的工程落点

后续真正生成的正式文件，不是停留在这份入口模板里，而是至少要落到下面这些工程位置之一：

| 对象 | 正式落点 | 为什么需要同步 |
|------|---------|---------------|
| 阶段验收正文 | `01_实验执行/[当前阶段]/阶段验收.md` | 这是当前阶段真正的正式放行文件 |
| 训练评估证据 | `experiments/[run_name]/`、`reports/tables/[文件].csv`、`reports/figures/[文件].png` | 训练评估型阶段需要物理证据支撑 `pass / partial / fail / not_applicable` |
| 治理闭环证据 | `b_class_auxiliary/coding_guards/[任务目录]/`、`reports/stage_reports/implementation_tracking/[当前阶段]/` | 治理汇总型阶段需要回链 guard 与阅读入口 |

**最小理解:**
- 本文件只是“选模板入口”,不是最终验收证据的存放位置
- 如果正式 `阶段验收.md` 写完了,却说不清它回链哪些 `run / report / guard`,说明这次验收还没有真正落地

## 6. 当前文件的角色边界

### 当前文件负责什么

- 负责把“阶段验收模板”从单一大模板拆成两个子模板
- 负责给执行者一个不会迷路的入口
- 负责说明两类模板分别适合什么阶段

### 当前文件不负责什么

- 不直接充当训练评估型阶段的正式填写模板
- 不直接充当治理汇总型阶段的正式填写模板
- 不代替 `模板3A_训练评估型阶段验收.md`
- 不代替 `模板3B_治理汇总型阶段验收.md`
