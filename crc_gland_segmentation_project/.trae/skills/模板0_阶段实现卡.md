# 模板0: 阶段实现卡(正式输出名 00_阶段实现卡.md)

> **使用场景**: 每次正式编码、正式配置修改或正式资产修改前,先把当前阶段到底要做什么、不做什么、为什么现在做这一步锁成一张正式卡片。
> **调用顺序**: 先看 `00_技能体系导航.md`,再走根目录 `crc-gland-research-alignment`,最后用本模板起草阶段卡正式输出名 00_阶段实现卡.md
> **与中央门禁的关系**: 本模板负责产出“阶段定义正文”; 真正的放行裁决必须继续由 `b_class_auxiliary/tools/check_stage_definition_gate.py` 生成阶段锁定门禁报告 stage_definition_gate_report.md

---

这份模板不是“研究阶段写一段摘要”。

它真正要回答的是:
- 当前阶段唯一目标到底是什么
- 当前阶段为什么现在做,而不是提前做下一个阶段
- 当前阶段允许改什么、不允许改什么
- 这些依据到底来自哪篇论文、哪个官方口径、哪个参考实现
- 这些依据最后会落到哪些工程对象
- 本轮打算用什么最小运行证据证明代码没写歪

**怎么写才算合格:**
- `已读文件` 不能只列文件名,必须说明为什么与当前阶段直接相关
- `为什么现在做这个,而不是下一个阶段` 不能省略,否则阶段边界仍然没有锁定
- `允许改 / 禁止改` 不能写成“本轮准备改哪些文件”,而是要写清协议边界
- `工程落点` 必须能回链到后面的 `Pre-check Guard` 与 `Post-QC Guard`
- `最小运行验证计划` 不能写成“后面再看”,要提前写明准备提取哪些真实物理证据

**红线提醒:**
- 不能把本模板写成研究阶段随手记事本
- 不能只说“参考了论文/仓库”,却不写章节、公式、commit 或文件落点
- 不能省略“为什么不是下一个阶段”
- 不能在 `阶段锁定状态 = blocked` 时继续进入正式编码
