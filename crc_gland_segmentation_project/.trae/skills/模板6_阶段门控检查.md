# 模板6: 阶段门控检查

> **使用场景**: 每次进入正式编码、正式配置修改或正式协议修订前,判断当前阶段是否真的允许开始。
> **调用顺序**: 先看 `00_技能体系导航.md`,并确保 `模板5_Pre-check提取.md` 已先完成
> **与中央门禁的关系**: 本模板负责回答“现在能不能开始”; 一旦这里结论是 `blocked`,中央 `crc-gland-coding-guard` 也必须停止后续实现与文档修改

---

这份模板不是“礼貌性写一句 allow”。

它真正要解决的是:
- 上一阶段到底有没有正式 `pass`
- 当前阶段要消费的资产是否真的已经准备好
- 本轮任务是否偷偷越过了当前阶段的边界
- 如果现在继续做,会不会破坏执行顺序和单变量原则

```markdown
# Stage Gate Check

## 1. 阶段信息
- 当前阶段:
- 上一阶段:
- 当前任务:

## 2. 上一阶段通过证据
- 通过文件:
- 通过状态:
- 关键交付物:

**这一节的正式依据通常来自:**
- `[上一阶段]/阶段验收.md`
- `b_class_auxiliary/coding_guards/[上一轮任务目录]/YYYYMMDD_[任务]_post_qc_guard.md`
- `00_总览与规范/00_执行导航.md`

不要只写“上一阶段应该已经完成”。要写清到底是哪个正式文件、哪个状态字段、哪个交付物在支撑本轮进入判断。

## 3. 当前阶段进入条件

| 条件 | 证据文件 | 检查方法 | 结果 |
|------|---------|---------|------|
41→| [阶段实现卡已存在且已写清当前阶段唯一目标] | `01_实验执行/[当前阶段]/00_阶段实现卡.md` | 检查 `## 3. 当前阶段唯一目标` 与 `## 5. 当前阶段允许改动 / 禁止改动` | `pass / fail` |
42→| [阶段锁定门禁已通过] | `b_class_auxiliary/coding_guards/[任务目录]/stage_definition_gate_report.md` | 打开文件确认 `stage_definition_gate_status = pass` | `pass / fail` |
43→| [上一阶段正式通过] | `[路径]` | 打开文件确认最终状态 | `pass / fail` |
44→| [当前阶段需要的交付资产已存在] | `[路径]` | 检查资产路径与字段 | `pass / fail` |
45→| [本轮任务没有越界到后续阶段] | `00_执行导航.md` / `[当前阶段总协议]` | 对照当前任务边界核对 | `pass / fail` |

**最小来源要求:**
- `证据文件` 不能只写“见上游文档”,要写出真实路径
- `检查方法` 不能只写“人工确认”,要写出看哪个字段、哪张表、哪个目录
- 如果本轮是模板治理,也要写明依据来自 `.trae/skills/[模板文件].md`、`b_class_auxiliary/coding_guards/[任务目录]/` 或 `01_实验执行/[阶段]/[正式协议].md`

## 4. 阻断项

| 阻断项 | 是否触发 | 说明 |
|-------|---------|------|
| `00_阶段实现卡.md` 缺失或仍是空壳 | `是/否` | [说明] |
| `stage_definition_gate_report.md != pass` | `是/否` | [说明] |
| 上一阶段未正式 `pass` | `是/否` | [说明] |
| 当前任务越界到后续阶段 | `是/否` | [说明] |
| 数据/评估/命名协议仍未冻结 | `是/否` | [说明] |
| 本轮拟修改文件不属于当前阶段允许范围 | `是/否` | [说明] |

## 5. 结论
- Stage Gate Result: `allow / blocked`
- 结论说明:

## 5.1 本轮允许进入的工程落点

| 对象 | 允许动作 | 预期落点 |
|------|---------|---------|
| 正式代码 | `create / update / not_applicable` | `src/[模块]/[文件].py` / `scripts/[脚本].py` / `tools/[脚本].py` |
| 配置与正式资产 | `create / update / not_applicable` | `configs/[配置].yaml` / `experiments/[run_name]/` / `reports/[子目录]/[文件]` |
| 模板与协议文档 | `create / update / not_applicable` | `.trae/skills/[模板文件].md` / `01_实验执行/[阶段]/[正式协议].md` |

**为什么要有这一节:**
- Stage Gate 不只是回答“能不能开始”,还要回答“开始后只能动哪些工程对象”
- 如果这里写不出预期落点,后面的 `Pre-check Guard` 和 `Post-QC Guard` 就无法核对是否越界
- 如果 `00_阶段实现卡.md` 和 `stage_definition_gate_report.md` 还没对齐,这里也不能写 `allow`

## 6. 红线提醒
- 如果 `Stage Gate Result = blocked`,本轮不能继续生成“正式修改版”文件
- 如果 `stage_definition_gate_report.md` 还是 `partial / fail`,这里不能嘴上写 `allow`
- 不能嘴上写 `allow`,但表格里关键条件其实有 `fail`
- `blocked` 不是坏事,它表示当前阶段发现了真实阻断项,避免把错误继续传到后面
```
