## 这个脚本的作用

这个脚本负责把 GlaS 数据集的原始图像和掩码，整理成 PyTorch 能直接消费的 `(image_tensor, mask_tensor, meta)` 三元组。

你可以把它理解成“数据入口总闸”。
后面的 `train.py`、`evaluate.py`、`visualize_samples.py` 都不是直接碰原始图片，而是先经过这里统一整理。

这一步为什么重要？
因为只要数据入口没统一，后面最容易出现三种混乱：
- 训练脚本和评估脚本各自写一套读取逻辑，最后 resize、归一化、mask 处理口径不一致。
- 你修了一个 bug，只修了 `train.py`，忘了 `evaluate.py`，结果训练和评估根本不是同一条数据链。
- 三个月后回来看，根本说不清“模型吃进去的到底是什么张量”。

## 这个脚本在整个阶段中的位置

如果你现在还不熟整个工程，可以先记住这一句：
`glas_dataset.py` 不负责训练模型，它只负责把“磁盘上的原始文件”变成“下游脚本能消费的标准输入”。

它在链路中的位置可以画成这样：

```text
splits/glas/*.csv
    + configs/glas.yaml
    + datasets/glas/images/
    + datasets/glas/masks/
            ↓
src/data/glas_dataset.py
            ↓
scripts/train.py
scripts/evaluate.py
tools/visualize_samples.py
```

**上游依赖：**
- `splits/glas/glas_train68.csv`、`glas_val17.csv`、`glas_testA.csv`、`glas_testB.csv`
- `configs/glas.yaml`
- `datasets/glas/images/` 与 `datasets/glas/masks/`

**下游消费者：**
- `scripts/train.py`：通过 `DataLoader(GlaSDataset(split="train"))` 读取训练样本
- `scripts/evaluate.py`：通过 `split="testA"` 或 `split="testB"` 读取正式测试样本
- `tools/visualize_samples.py`：直接调用 `__getitem__()` 导出三联图

**如果这里写错了，会连带影响什么：**
- resize 错了，训练和评估都会错。
- mask 插值错了，loss 和指标都会一起被污染。
- meta 返回删掉了，后续误差分析和可视化回溯会直接断链。

## 脚本核心逻辑

### 主要流程

当 `train.py` 调用 `GlaSDataset(split="train")` 时，内部主链路通常是下面这几步：

1. **初始化阶段**
   - 读取 `splits/glas/glas_train68.csv`
   - 解析当前 split 的图像文件名和掩码文件名
   - 读取 `configs/glas.yaml` 中与数据有关的字段
   - 初始化图像和 mask 的同步变换

2. **单样本读取阶段**
   - 从 `datasets/glas/images/` 打开 RGB 图像
   - 从 `datasets/glas/masks/` 打开灰度掩码
   - 把图像和掩码 resize 到统一尺寸
   - 图像走归一化，掩码走二值化
   - 返回 `(image_t, mask_t, meta)`

3. **下游消费阶段**
   - `DataLoader` 负责 batch 化
   - 模型只拿到 tensor
   - 分析工具额外读取 `meta["image_path"]` 和 `meta["mask_path"]`

### 关键规则 1：图像和掩码必须同步变换

这句话听起来像常识，但很多新手第一次写分割数据集时就是死在这里。
如果图像做了 resize / flip / crop，而 mask 没同步做同样变换，那么模型看到的输入和监督标签就不是同一个空间坐标系。

**当前实现意图：**
- 图像和掩码共用同一套空间变换
- 但图像和掩码的插值策略不同

> **溯源锚点：**
> - 理论依据: `[Ronneberger et al., 2015] §2 Data Augmentation`
> - 当前实现: `src/data/glas_dataset.py` → `__getitem__()`
> - 冻结表对应: `02_参数冻结总表.md` 中的 `data.resize_dim`

### 关键规则 2：mask 必须走 NEAREST，不能走 BILINEAR

用人话说就是：
mask 是离散标签，不是连续灰度图。
你如果拿双线性插值去缩放，边界上的 `0/1` 会被抹成 `0.3/0.7` 这种中间值。
训练时 loss 会学到一堆“软标签”，但评估又按硬标签算，最后最容易出现“loss 看着正常，Dice 却很怪”的现象。

示意写法：

```python
image = image.resize((512, 512), resample=Image.BILINEAR)
mask = mask.resize((512, 512), resample=Image.NEAREST)
mask_binary = (mask_np > threshold).astype(np.float32)
```

> **溯源锚点：**
> - 理论来源: `[Sirinukunwattana et al., 2017] §2.2`
> - 冻结回链: `02_参数冻结总表.md` 中 `data.mask_threshold = 0.5`
> - 当前实现: `src/data/glas_dataset.py` 第 `[示例行号]`

### 关键规则 3：返回 meta 不是锦上添花，而是回溯锚点

很多人第一次写 Dataset 时只返回 `image, mask`。
如果只是为了训练，这样勉强能跑。
但一旦你要做错误分析、可视化导出、失败样本回查，没有 `meta` 你就只剩下 batch index，根本不知道是哪一张图出了问题。

因此更稳的设计是：
- `image_t` 负责给模型吃
- `mask_t` 负责给 loss 用
- `meta` 负责给人类回查

### 为什么不用别的设计？

| 候选方案 | 优点 | 缺点 | 最终决策 |
|---------|------|------|---------|
| 直接在 `train.py` 里写读取逻辑 | 简单直接 | 训练、评估、可视化各写一遍，极易漂移 | 否决 |
| 单独 `dataset.py` 封装 | 单一入口，改一处全链生效 | 多一层跳转 | 采用 |
| 直接继承通用 dataset 模板 | 看起来规范 | GlaS 的 split / meta / mask 处理仍要大量改 | 否决 |

## 误区和排错点

### 误区 1：掩码也能用双线性插值

这是最典型的坑。
图像可以双线性，mask 不可以。
否则边界标签会被污染。

### 误区 2：GlaS 没有显式 patient_id，所以随便重划 split 也行

不行。
这个项目追求的不是“自己内部能跑”，而是“能和官方口径与论文口径对齐”。
你一旦自己重划 split，后面的数字就不再可比。

### 误区 3：为了简化接口，把 `meta` 删掉

短期看会更干净，长期看会直接切断误差分析、导出命名和可视化回溯。

### 协议违规风险

- 如果你把 `resize_dim` 从 `512` 改成 `256`，但没有先回到冻结表改协议，这就是违规，不是“优化”。
- 如果你把 `testA/testB` 合并成一个统一测试入口，后面评估口径会和官方拆分口径冲突。
