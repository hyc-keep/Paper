# 制度完成定义

## 正式角色

- 当前文件名: `制度完成定义.md`
- 本文件只负责定义“什么叫完成”
- 本文件不负责代替根目录总入口做路由
- 本文件必须把口头词裁成可审计门禁词
- 凡涉及“什么叫 `pass / partial / fail`”以及“什么时候允许进入下一步”,本文件的解释权高于项目内导航

## 边界/分层

本文件只定义“什么叫完成”,不直接替代:

- 根目录总入口 `d:\12_Medical_Image_Segmentation\Paper\.trae\skills\crc-gland-coding-guard\SKILL.md`
- 项目内导航 `d:\12_Medical_Image_Segmentation\Paper\crc_gland_segmentation_project\.trae\skills\00_技能体系导航.md`
- 学习型说明文规程 `d:\12_Medical_Image_Segmentation\Paper\crc_gland_segmentation_project\.trae\skills\实验执行文档编写规范.md`

它只负责把这些对象里出现的“完成”“可写”“可放行”裁成零歧义门禁。

## 状态口径

- `Stage Gate`: `allow / blocked`
- `Diagnostics / Post-QC`: `pass / partial / fail / not_applicable`
- `implementation_tracking`: `not_started / blocked / partial / pass`

不允许把这三套状态混成一句“差不多完成了”。

## 门禁/审查

当前文件的门禁审查职责只有两件事:

1. 把“可进入下一步”裁成必须存在的正式文件与 gate 字段
2. 阻止任何没有正式依据、没有正式产物、没有真实物理证据的口头放行

任何“当前该去哪个阶段”或“现在允许进入下一步”的口头裁决,都必须同时给出本轮已遍历的 skill / 导航 / 规程 / 当前阶段正式文件清单。

如果没有显式已读清单,则该路由结论只算未完成判断,不能视为正式放行意见。

## 唯一原则

- 缺少真实依据、真实边界或真实物理证据,就不能宣称完成
- 缺少正式产物,就不能口头放行到下一步
- 正式代码、配置或正式资产一旦开始改动,就必须同步留下实现依据留痕,不能等到说明文阶段再回忆
- 文档只能解释已经被代码和证据证明成立的事实
- 当前仍在治理,或当前编号阶段正式产物 / gate 还没收口时,不能把下一编号阶段写进当前结论

## 文件三分归属（适用所有阶段）

本节定义“一个文件属不属于项目”。任何阶段产生的任何文件,必须被归入且只归入下面三类之一,不允许出现“既算项目又算辅助”的混合件。

### A. 项目正式对象（交付 + 写说明文）

满足任一即归 A:

- 当前阶段计划文件的「代码落点 / 代码落地接口 / 正式资产 / 产物清单」明确点名的脚本、模块、配置或产物
- 为满足计划某条明确要求所必需,且能在 `实现依据记录.md` 里回链到具体计划条款号或 `结直肠腺体分割_正式参考资料` 出处

归入 A 的硬要求: 必须写出客观依据(计划条款号 或 参考资料出处),不允许凭“感觉有用”归入 A。

### B. AI 辅助与流程文件（不交付 + 不写说明文 + 交付时删除）

满足任一即归 B:

- skill 门禁与治理: 门禁/治理类 `check_*`、`enforce_*`、`*gate_report*`、`b_class_auxiliary/`、`.trae/skills/`
- 计划未点名、仅为生成某个计划产物而临时多造的工具脚本
- 诊断、审稿回填、runtime 中间件等只服务本轮流程、删掉不影响项目产物链运行的文件

当前项目的 B 类专用物理目录固定为 `crc_gland_segmentation_project/b_class_auxiliary/`。
其中 `tools/`、`runtime_checks/`、`coding_guards/`、`workflow_gates/` 这类 B 类工具实现、运行留痕与门禁产物统一落在该目录下,避免继续与 `tools/` 主区或 `reports/stage_reports/` 等 A 类正式对象混放。

例外说明:

- `.trae/skills/` 仍是固定路径的 B 类规程层,不搬入 `b_class_auxiliary/`
- B 类 gate/runtime/check/enforce 脚本的真实实现统一放在 `b_class_auxiliary/tools/`
- 根目录 `tools/` 只保留 A 类正式工具与 C 类待裁决工具,不再承载任何 B 类 gate/runtime 脚本
- A 类正式工具按编号阶段进入 `tools/stageXX_[阶段名]/`
- C 类待裁决工具统一进入 `tools/c_pending_review/`

### C. 存疑待清理（暂留 + 交付前复审）

- 计划没点名,常识上也无法确定“一个正常严谨的研究者做本项目会不会放它”的文件
- C 类必须在 `实现依据记录.md` 显式登记,等交付前清理时逐个裁决转入 A 或 B

### 三分归属红线

- 一个文件只能属于 A / B / C 之一,禁止把“计划要求的项目内容”与“AI 辅助检查内容”写进同一个文件
- 如果一段 AI 辅助内容确有保留价值,必须单独另存为一个 B 类文件,不允许塞进任何 A 类项目文件里
- 实现中若新增计划未点名的文件,必须当轮在 `实现依据记录.md` 显式登记一句: 「此文件计划未点名,归类 = B 或 C,理由 = XX」
- 未显式登记归类的计划外文件,一律视为未完成,不允许进入总放行
- A 类清单只能收录有客观依据的项目对象;发现被混入的 B/C 文件必须移出 A 类清单

`workflow_gate_report.md` 不是阶段锁定完成的定义件。

### 3. Pre-check 完成

### 4. 代码质量成立

| 项目 | 定义 |
| --- | --- |
| 必要输入 | 本轮涉及正式代码、配置或正式资产改动,且 runtime 检查已运行 |
| 必需文件 | `实现依据记录.md`、`runtime_check_report.md`、`runtime_evidence.json`、`runtime_check.log`、`code_quality_gate_report.md` |
| 判定字段 | 默认按完整训练画像裁决: `code_quality_gate_report.md = pass`, `实现依据记录.md` 已把每个被改动的正式文件逐文件写清计划条款、论文/参考代码、冻结参数和工程收敛决定,且 `runtime_evidence.json` 至少包含 `sample_path / sample_id / input_shape / input_dtype / target_shape / target_dtype / target_unique_values / output_shape / output_dtype / loss_value / loss_is_finite / backward_executed / optimizer_step_executed` |
| 禁止口头放行情形 | 只有 raw sample 可读、只有 markdown 报告、只有“我参考过一些资料”的口头说法、仍存在 placeholder / stub / `NotImplementedError` |
| 通过后允许去哪一步 | `总放行` |

缺少任一证据文件,或任一关键状态不是 `pass`,都不允许宣称“代码已成立”。

当前只允许一个明确例外:

- 如果当前编号阶段=`01_数据协议`,且本轮正式改动只服务 `train entrypoint preflight`,则 `代码质量成立` 允许按 `data_protocol_preflight` 画像裁决
- 这时仍然要求 `实现依据记录.md`、`runtime_check_report.md`、`runtime_evidence.json`、`runtime_check.log`、`code_quality_gate_report.md` 全部存在
- 这时 `runtime_evidence.json` 至少要给出 `sample_path / sample_id / input_shape / input_dtype / target_shape / target_dtype / target_unique_values / asset_manifest / data_stage_pass / handoff_ready / split_asset_exists / data_config_registered / entrypoint_check_pass`
- 这时 `loss_value / loss_is_finite / backward_executed / optimizer_step_executed` 允许为 `not_applicable` 或 `null`,但 `runtime_check_report.md` 与 `code_quality_gate_report.md` 必须显式写明 `runtime_profile = data_protocol_preflight`
- 这种 `pass` 只代表 `01_数据协议` 所需的正式数据链与训练入口预飞成立,不代表完整训练链已经成立,也不允许被拿去替代 `02_UNet流程验证` 及后续模型阶段的完整训练代码质量结论

交付前清理只在准备交付时执行,不是每轮编号阶段的必经步骤。

## 红线

- 不允许把 `allow / blocked`、`pass / partial / fail`、`implementation_tracking` 混成同一层判断

- 不允许把“读过一些文件”冒充成“研究完成”
- 不允许把“阶段卡写出来了”冒充成“阶段锁定完成”
- 不允许未显式展示本轮已遍历清单,就给出“当前已经到哪一步”的正式判断
- 不允许把原始样本可读性冒充成正式训练链成立
- 不允许把长文档、漂亮文档、复杂状态字段当成代码质量证明
- 不允许把编码依据留痕拖到学习型说明文阶段才补
- 不允许把多个被改动的正式 `py` 文件压成一条无法逐文件回查的实现依据
- 不允许没有 `workflow_gate_report.md` 就宣称正式任务通过
- 不允许当前治理或当前编号阶段未收口,却先提下一编号阶段
- 不允许把“计划要求的项目内容”和“AI 辅助检查内容”混进同一个文件
- 不允许把计划未点名、未在 `实现依据记录.md` 登记归类的文件塞进 A 类交付清单
- 不允许给 B 类(AI 辅助与流程文件)写逐文件交付说明文
