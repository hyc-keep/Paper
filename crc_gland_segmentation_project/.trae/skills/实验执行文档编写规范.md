# 实验执行文档编写规范(implementation_tracking专用规程)

## 目的

本规范只服务一个目录:

`d:\12_Medical_Image_Segmentation\Paper\crc_gland_segmentation_project\reports\stage_reports\`

它的作用不是当总入口,而是告诉你学习型说明文该怎么写才不会又空又假。

## 正式角色

- 只负责 `README.md`、`implementation_status.md`、`真实对象文件名.md`、`当前阶段为什么能pass以及下一步怎么看.md`
- 使用前先看 `00_技能体系导航.md`
- 正式模板与 Guard 留痕统一回到 `模板1~10`
- 当前阶段如果还没有代码证据,不要先来写这里

写前必须依次读:

1. `crc_gland_segmentation_project/.trae/skills/00_技能体系导航.md`
2. `crc_gland_segmentation_project/.trae/skills/TCGA原始标杆对齐清单.md`
3. 至少 2 份真实 TCGA 原始学习说明文
4. `crc_gland_segmentation_project/.trae/skills/示例_学习型说明文_融合版.md`
5. `crc_gland_segmentation_project/.trae/skills/学习型说明文人工审稿清单.md`

如果本轮对应的是正式代码、配置或正式资产改动,写说明文前还必须能回链:

- 研究定标记录.md
- research_alignment_gate_report.md
- 实现依据记录.md
- 阶段卡正式输出名 00_阶段实现卡.md
- 阶段锁定门禁报告 stage_definition_gate_report.md
- `模板R_研究定标记录.md`
- `模板I_实现依据记录.md`
- `模板0_阶段实现卡.md`
- `b_class_auxiliary/tools/check_research_alignment_gate.py`
- `b_class_auxiliary/tools/check_stage_definition_gate.py`

## 当前边界

- 这份文件不替代 `论文实验项目文档编写规范_完整版.md`
- 这份文件不替代 `模板1~10`
- 这份文件只解释已经成立的代码、配置、结果和资产
- `Pre-check Guard`、`Post-QC Guard`、`diagnostics_result.txt` 不是学习型说明文本体

## 交付范围优先规则

学习型说明文默认只覆盖 `制度完成定义.md`「文件三分归属」里判定为 **A 类(项目正式对象)** 的文件。
这里的“项目有关”不能再按“本轮碰过哪些文件”或“看起来以后会不会一起交付”来主观判断。

先分清三类对象:

- `A 类(项目正式对象)`: 要交付、要写说明文
- `B 类(AI 辅助与流程文件)`: 不交付、不写说明文
- `C 类(存疑待清理)`: 暂留,交付前复审; 在裁决前不写正式交付说明文

判定口径以 `制度完成定义.md` 为唯一权威:

- 只有 A 类对象才进入 `正式对象 -> 对应说明文` 映射
- B 类对象一律不进入学习型说明文覆盖范围
- C 类对象在交付前清理裁决前,只允许在 `实现依据记录.md` 保留归类登记,不进入正式交付说明文

A 类不能靠“感觉会交付”来判,必须有客观依据:

- 是当前阶段计划(`结直肠腺体分割_plan_优化版`)的「代码落点 / 代码落地接口 / 正式资产 / 产物清单」明确点名的对象
- 或为满足计划某条明确要求所必需,且能在 `实现依据记录.md` 回链到计划条款号或 `结直肠腺体分割_正式参考资料` 出处

固定不纳入学习型说明文覆盖范围的对象:

- `.trae/skills/**`
- `b_class_auxiliary/coding_guards/**`
- `Pre-check Guard`、`Post-QC Guard`
- `diagnostics_result.txt`
- 门禁/治理类 `check_*`、`enforce_*`、`*gate_report*` 等流程脚本
- 计划未点名、仅为生成某个计划产物而临时多造的工具脚本
- 任何只服务 runtime / code-quality / workflow gate 或审稿回填的内部留痕

默认优先检查的 A 类候选对象:

- `src/`、`scripts/`、`configs/`、`datasets/`、`splits/`、`experiments/`、`external/` 下,且能被计划或 `实现依据记录.md` 客观回链的正式对象
- 被当前阶段协议或 `结直肠腺体分割_plan_优化版` 明确点名的 `tools/`
- 被当前阶段协议明确要求作为正式交付结果保留的 `reports/`

动态裁定方法:

1. 每次进入 `说明文档` 阶段,先根据当前 turn 的正式改动文件、当前阶段协议和 `实现依据记录.md`,按 A / B / C 三分类逐个判定
2. 只对 A 类对象建立 `正式对象 -> 对应说明文` 映射
3. B 类与 C 类对象也要在清单里显式列出归类理由,但不能冒充项目说明文对象
4. 判断标准不是“这个文件本轮是否出现过”,而是“它是否有客观依据进入 A 类项目正式对象”

### 写文档前的强制对象清单

正式开始写 implementation_tracking 文档前,必须先在工作记录或 Post-QC 回链上下文里明确列出:

| 字段 | 必填内容 |
| --- | --- |
| 对象路径 | 例如 `scripts/train.py`、`src/data/datasets.py`、`configs/data/glas.yaml` |
| 归类 | `A / B / C` |
| 归类依据 | `计划条款号 / 参考资料出处 / 归 B 或归 C 的理由` |
| 对象类型 | `python` / `yaml` / `csv` / `image` / `formal_md` |
| 是否首次正式可用 | `yes / no` |
| 对象级说明文 | A 类对象填写如 `scripts_train.py.md`; B/C 类填写 `not_applicable` |
| 入口同步项 | 如有阶段入口变化,填写单个真实阶段入口文档路径,例如 `reports/stage_reports/implementation_tracking/01_数据协议/README.md`; 无则 `not_applicable` |

没有这张“本轮交付范围内正式对象清单”,不得宣称“本轮项目文件说明已完成”。
如果清单里存在新的 A 类 Python / YAML / CSV / 结果资产,但没有对应逐文件说明文,也不得把入口文档当成替代品。

## 状态口径边界

- `Stage Gate`: `allow / blocked`
- `Diagnostics / Post-QC`: `pass / partial / fail / not_applicable`
- `implementation_tracking` 阅读入口: `not_started / blocked / partial / pass`

固定规则:

- `README.md` 和 `implementation_status.md` 只能用 `not_started / blocked / partial / pass`
- 不要把 `fail / not_applicable` 直接当阶段总体状态
- 一旦已经出现正式 Guard、正式代码改动或正式资产,就不能继续写 `not_started`

## 核心理念

学习型说明文默认要同时满足两条线:

1. 小白真的能顺着文档读懂
2. 论文级依据和物理证据可以回查

最少要写清 7 件事:

1. 这个对象在整个阶段里干什么
2. 为什么这样设计,不是别的设计
3. 主流程怎么走
4. 关键公式或原理来自哪里
5. 上游和下游是谁
6. 最容易误解什么
7. 读完后怎么自检

## 结构化溯源卡片

`source_file.py.md` 默认要有 `## 结构化溯源卡片`。

最小骨架如下:

```markdown
## 结构化溯源卡片

- 正式对象: `src/...` / `scripts/...` / `tools/...`
- 对应阶段: `XX_阶段名`

### 论文依据
- 论文: [论文名, 年份]
- 章节: [第X节]
- 公式/定义: [公式(N) / 定义 / 页码]

### 代码依据
- 仓库: [GitHub URL]
- 文件: [相对路径]
- commit: [哈希]
- 许可证: [MIT / Apache-2.0 / ...]

### 冻结回链
- 冻结文件: `结直肠腺体分割_plan_优化版/01_实验执行/00_总览与规范/02_参数冻结总表.md`
- 对应字段: `[字段1, 字段2]`

### 当前实现落点
- 文件: `src/...`
- 符号: `ClassName / function_name()`
```

只写“参考了某论文/某仓库”但没有章节、公式、commit、文件落点,视为不完整。

## 文档组最小结构

一个阶段的 `implementation_tracking` 文档组最少建议保留:

- `README.md`
- `implementation_status.md`
- 至少 1 份逐文件说明文（例如 `scripts_train.py.md`）
- `当前阶段为什么能pass以及下一步怎么看.md`

如果本轮新增了正式对象,就要在对应说明文里写出:

- 路径
- 输入输出
- 关键字段或关键张量
- 验证步骤
- 当前真实结果

## 八、强制执行机制

### 1. 学习型说明文写前条件

只有当下面条件成立时,才允许正式写文档:

- 阶段锁定门禁报告 stage_definition_gate_report.md 为 `pass`
- `实现依据记录.md` 已存在并且不是占位壳
- `runtime_check_report.md` 已存在
- `code_quality_gate_report.md` 已有结论
- `Post-QC Guard` 已能说明本轮改动
- 已列出“本轮交付范围内正式对象清单”,并完成 `正式对象 -> 对应说明文` 映射

### 2. 自动门禁

写完后必须运行:

`python b_class_auxiliary/tools/check_learning_docs.py --precheck-guard [Pre-check Guard相对路径] --post-qc-guard [Post-QC Guard相对路径]`

如果本轮同时修改了 `.trae/skills/*.md` 或 `01_实验执行` 正式协议,还必须运行:

`python b_class_auxiliary/tools/check_formal_stage_docs.py --post-qc-guard [Post-QC Guard相对路径]`

### 3. 人工终审

命中新建逐文件说明文（例如 `scripts_train.py.md`）、重写主体解释、或新写验收说明文时,还要执行:

- `crc_gland_segmentation_project/.trae/skills/学习型说明文人工审稿清单.md`
- 并回看 `crc_gland_segmentation_project/.trae/skills/TCGA原始标杆对齐清单.md`

## 5. 最短选择指南

- 如果你要写入口说明: 先写 `README.md`
- 如果你要写阶段现状: 先写 `implementation_status.md`
- 如果你要解释脚本或资产: 写“真实对象文件名 + .md”形式的逐文件说明文，例如 `scripts_train.py.md`
- 如果你要解释当前为什么能放行: 写 `当前阶段为什么能pass以及下一步怎么看.md`

## 红线(触发立即回退)

- 代码还没给出真实物理证据,就先补长文
- 没有 `实现依据记录.md`,却试图在说明文里反向补“当时参考了什么”
- 只有口语化,没有来源锚点和验证步骤
- 只有篇幅,没有路径、字段、数值或命令
- 把 Guard 留痕当成学习型说明文
- 它不是在定义 `01_实验执行/[阶段名]/` 主协议目录的必备文件清单
- 如果你要写阶段主协议、设计依据、阶段验收或 guard 留痕,回到 `模板1~10`
- 没有先列“本轮交付范围内正式对象清单”,就开始写“项目说明已完成”
- 只写阶段入口页,却没有补新增正式 Python / YAML / CSV / 资产的逐文件说明

### 学习型说明文的强制映射规则

后续不能只写一篇总说明,然后默认“已经覆盖所有代码改动”。

必须按下面规则,对**交付范围内正式对象**建立 `正式对象 -> 对应说明文` 映射:

| 正式对象 | 对应说明文 | 最低要求 |
|---------|-----------|---------|
| Python 脚本 / 模块 | `相对路径扁平化 + .py.md` | 首次正式可用时必须创建 |
| 配置文件（YAML） | `相对路径扁平化 + .yaml.md` | 首次正式进入流程时必须创建 |
| 结果/资产文件（CSV/图像/MD清单） | `相对路径扁平化 + 原始后缀 + .md` | 首次成为正式交付物时必须创建 |
| 阶段文档组入口变化 | `README.md` + `implementation_status.md` | 文件增删、阅读顺序或阶段结论变化时必须同步更新 |

补充硬规则:

- `README.md` 和 `implementation_status.md` 只负责导航、阶段现状和阅读顺序,不能替代真实工程对象的逐文件解释
- 只要 `scripts/`、`src/`、`configs/`、`splits/`、`datasets/` 中出现新的 A 类对象,就必须建立对应说明文; `tools/` 和 `reports/` 只有在计划点名或 `实现依据记录.md` 可客观回链时才允许纳入 A 类
- “本轮碰过了某文件”不等于“这个文件值得写说明文”; 只有“这个文件有客观依据进入 A 类”才等于必须纳入映射
- “本轮写了入口 README”不等于“代码说明完成”; 只有逐文件映射补齐后才允许这么说

这里的 `正式对象` 默认指:

- `制度完成定义.md`「文件三分归属」里判定为 A 类的对象
- 并且能被当前阶段协议、`结直肠腺体分割_plan_优化版` 或 `实现依据记录.md` 客观回链的对象

这里默认不包括:

- skill 体系文件
- guard 留痕
- 内部 gate 报告
- 只为当前开发流程服务、最终不会跟项目一起上传的文件

**怎么判断“首次正式可用”:**
- 不再是 TODO/占位壳
- 上游输入和下游输出都已经明确
- 后续其他人真的需要靠它理解、运行或验收

### 必须包含的章节

```markdown
# XX_阶段名称 阅读入口

## 先看结论

- 当前 `XX_阶段名称` 这个阶段的状态: [not_started/blocked/partial/pass]
- 这里的 `pass` 指的是: [用一句人话说明"通过"的物理含义]
- 这里的 `pass` 不指: [明确说明哪些不算通过条件]
- 当前真正已经形成闭环的是:
  - [列出关键资产1]
  - [列出关键资产2]
  - ...

## 最推荐阅读顺序

请直接按下面顺序读:

1. `01_先看这一组文档怎么读.md`
2. `02_当前阶段到底已经生成了什么正式资产.md`
3. ...
N. `最后_当前阶段为什么能pass以及下一步怎么看.md`

## 这一版和旧版最重要的区别

[说明本次重写解决了什么问题,为什么要这样组织]

## 你现在最该先搞明白的四件事

- [关键理解点1]
- [关键理解点2]
- [关键理解点3]
- [关键理解点4]

[用一段话说明理解这四件事后能解决什么问题]
```

**README 的最小质量线:**
- 不能只罗列文档顺序,至少要让读者看懂“这个阶段为什么已经能看/为什么现在还不能误判”
- 至少保留 2 组口语化解释信号（例如“你现在不用再...”“换句话说...”“你可能会问...”）
- 至少保留 1 处诚实边界或误区提醒（例如“这里的 `pass` 不指...”“不要误会...”）
- 至少保留 1 处联读收口提示（例如“后面进入下一阶段前先看什么”）
- 每个 `##` 章节开头 3 行内至少先给一句结论或 1 条 bullet
- 连续纯正文不要超过 `8` 行；如果解释太长，就拆成短段或对照结构

  - [资产1]: 路径 `xxx`, 作用 [xxx]

## 当前已经实现的脚本

[列出真正有实质内容的脚本]

## 当前仍然是占位的脚本

[诚实列出还没实现的脚本,避免误导]

**implementation_status 的最小质量线:**
- 不能只写状态枚举,至少要把“当前最真实的说法”和“为什么这样组织阅读入口”讲清楚
- 至少保留 2 组口语化解释信号（例如“如果你只想最快理解...”“换句话说...”“你现在可能会问...”）
- 至少保留 1 处诚实边界提示（例如“不是所有...都已完成”“这两句话不冲突”）
- 至少保留 1 处联读收口提示（例如“最短路径是...”“下一个阶段前先确认...”）
- 每个 `##` 章节开头 3 行内至少先给一句结论或 1 条 bullet
- 连续纯正文不要超过 `8` 行；如果内容变长，优先拆成“已完成 / 未完成 / 下一步”这类对照块

## 十、学习型说明文的触发时机规则

### 核心原则

学习型说明文（例如 `scripts_train.py.md`）的写作不是"每次改一行代码就写一份"，也不是"全部做完再补"。
正确的触发点是：**脚本/模块首次达到「可独立运行」状态时**。

### 什么叫「可独立运行」

满足以下全部条件：
- 脚本能被 `python xxx.py` 正常执行（不报 ImportError / SyntaxError）
- 输入和输出已明确（不是 TODO 占位）
- 核心逻辑已实现（不是只有空函数壳）

### 触发时机速查表

| 场景 | 是否需要写说明文 | 说明 |
|------|----------------|------|
| 脚本首次达到可独立运行状态 | **必须写** | 这是说明文的"出生点" |
| 对已有脚本的小 bug 修复（不改接口/不改逻辑流） | **不需要重写整篇** | 但必须在已有说明文末尾追加一行版本记录 |
| 对已有脚本新增功能分支 | **更新已有说明文** | 在对应章节补充新分支逻辑，不另起新文件 |
| 对已有脚本重构（改了接口/改了核心流程） | **重写对应章节** | 标注重构原因和新旧差异 |
| 新增配置文件/数据文件 | **写对应的资产说明文** | 触发点同上：文件首次进入「正式可用」状态 |
| 阶段结束时 | **总 review** | 检查所有说明文是否覆盖齐全，补齐遗漏 |

**补一条硬规则:**
- 只要代码/配置/资产发生了正式变更,就必须在 `Post-QC Guard` 的文档映射表里写明：本轮是“新建说明文 / 更新说明文 / 仅追加版本记录 / not_applicable”
- 如果这里说“仅追加版本记录”,对应说明文里就真的要出现 `## 版本更新记录`

### 版本追加记录格式

当小修不需要重写时，在说明文末尾追加：

```markdown
## 版本更新记录

| 日期 | 改动内容 | 影响范围 | 是否需要重新验证 |
|------|---------|---------|----------------|
| YYYY-MM-DD | [一句话描述改动] | [影响了哪个函数/哪个输出] | 是/否 |
```

### 禁止行为

- ❌ 脚本还是占位壳就提前写说明文（没有实质内容可讲）
- ❌ 阶段做完后才"统一补齐"所有说明文（容易敷衍、遗忘细节）
- ❌ 每次 1 行改动就更新说明文（制造噪音、浪费精力）

## 十一、量化深度锚点（最小体量标准）

### 为什么需要量化锚点

本项目的学习型说明文需要同时承载「TCGA 原始标杆的小白友好」和「论文溯源」两层内容，
再叠加当前项目额外要求的视觉层次与质检透明度。没有量化标准时，AI 容易写得太短（敷衍）或太长（堆砌）。

### 代码脚本说明文档（例如 `scripts_train.py.md`）最小体量

| 对象类型 | 最小正文行数 | 说明 |
|------|---------|------|
| 核心模块 / 主流程脚本（通常对应 `src/`、`scripts/`） | ≥ 120 行（不含空行和纯代码块） | 低于此值通常说明原理、衔接、误区讲得不够透 |
| 工具脚本 / 薄包装脚本（通常对应 `tools/` 或单一功能入口） | ≥ 80 行（不含空行和纯代码块） | 允许更短，但仍必须讲清输入/输出/验证 |

| 维度 | 最小要求 | 说明 |
|------|---------|------|
| 结构化溯源卡片 | 必填 | 论文/章节或公式/仓库/commit/许可证/当前实现落点至少覆盖应有字段 |
| 溯源锚点 | ≥ 3 处 | 论文/冻结表/参考仓库，至少覆盖 3 个不同来源 |
| 误区条目 | ≥ 2 条 | "新手最容易误解为..." 类内容 |
| 自检任务 | ≥ 1 个可执行操作 | 5~10 分钟内可完成的验证动作 |
| 协议违规风险提示 | ≥ 1 条 | "如果你改了这个参数，会违反冻结表§X" |
| 参数与冻结表对照 | ≥ 1 处 | 至少标注一个关键参数的冻结依据 |
| 连续纯正文上限 | ≤ 10 行 | 超过后应拆成短段、列表、对照表或流程链 |
| 主章节结论先行 | ≥ 4 个章节命中 | 至少 4 个 `##` 章节在开头 3 行内先给结论句或总结 bullet |

### 资产文件说明文档（例如 `splits_glas_glas_train68.csv.md`）最小体量

| 对象类型 | 最小正文行数 | 说明 |
|------|---------|------|
| 正式结果/统计资产（CSV、报告、图像资产） | ≥ 60 行（不含空行和纯代码块） | 低于此值说明字段解释或验证步骤不够详细 |
| 薄配置资产（字段较少的 YAML、清单型 MD） | ≥ 50 行（不含空行和纯代码块） | 允许更短，但仍必须包含真实示例、字段解释和验证步骤 |

| 维度 | 最小要求 | 说明 |
|------|---------|------|
| 真实数据示例 | ≥ 1 段 | 必须展示当前项目中的真实内容，禁止用抽象占位 |
| 逐字段解释 | 覆盖率 100% | 每个字段/列都必须解释，不能遗漏 |
| 手工验证步骤 | ≥ 1 组 | 包含具体路径 + 期望结果 |
| 误区条目 | ≥ 1 条 | 至少说明一个容易误读的点 |
| 口语化解释信号 | ≥ 2 组 | 至少出现“你现在可能会问 / 用人话说 / 你可以把它理解成 / 换句话说”等引导 |
| 联读或收口提示 | ≥ 1 处 | 至少告诉读者下一步看什么,或学完后该具备什么判断力 |
| 连续纯正文上限 | ≤ 8 行 | 超过后应拆成字段块、验证步骤或对照表 |
| 主章节结论先行 | ≥ 3 个章节命中 | 至少 3 个 `##` 章节在开头 3 行内先给结论句或总结 bullet |

### 验收说明文档（`当前阶段为什么能pass以及下一步怎么看.md`）最小体量

| 维度 | 最小要求 | 说明 |
|------|---------|------|
| 正文行数 | ≥ 80 行 | 必须充分解释通过逻辑 |
| 物理证据 | ≥ 3 处 | 文件路径 + 具体字段/数值 |
| 回退条件 | ≥ 1 条 | 什么情况下必须回退重做 |
| 下游放行前置条件 | ≥ 2 条 | 下一阶段进入前必须确认什么 |
| 口语化解释信号 | ≥ 2 组 | 至少出现“你现在可能会问 / 用人话说 / 说白了 / 换句话说”等解释句式 |
| 风险或误区提示 | ≥ 1 条 | 至少说明一个“为什么现在不能误判成 pass”的坑点 |
| 连续纯正文上限 | ≤ 8 行 | 超过后应拆成条件块、证据表或回退清单 |
| 主章节结论先行 | ≥ 3 个章节命中 | 至少 3 个 `##` 章节在开头 3 行内先给结论句或总结 bullet |

### 阶段入口文档（`README.md` / `implementation_status.md`）最小体量

| 维度 | 最小要求 | 说明 |
|------|---------|------|
| 正文行数 | ≥ 40 行 | 低于此值通常说明阅读入口仍然太像目录标签 |
| 口语化解释信号 | ≥ 2 组 | 至少出现“如果你只想最快理解 / 换句话说 / 你现在不用再 / 当前最真实的说法”等 |
| 诚实边界提示 | ≥ 1 条 | 至少出现“不指 / 不等于 / 还没有 / 不冲突 / 不要误会”等 |
| 联读收口提示 | ≥ 1 条 | 至少说明“下一步看谁 / 最短路径怎么读 / 进入下阶段前先确认什么” |
| 连续纯正文上限 | ≤ 8 行 | 超过后应拆成短段、编号链或对照块 |
| 主章节结论先行 | ≥ 3 个章节命中 | 至少 3 个 `##` 章节在开头 3 行内先给结论句或总结 bullet |
| 结构化阅读信号 | 编号链 ≥ 3 或表格行 ≥ 2 | 至少有一类明显骨架让读者快速定位 |

### 超出上限的处理

如果某份文档写到 300+ 行仍不能收尾，说明该脚本职责过重，应考虑拆分脚本或拆分说明文为 Part A / Part B。

### 如何用这个锚点

- 写完后数一下正文行数（VSCode 底栏可直接看）
- 搜索 `溯源锚点` / `论文` / `冻结表` 关键词，数一下出现次数
- 搜索 `误区` / `容易误解` / `掉坑`，确认条目数量
- 如果不达标，不能标记为完成

## 最后的话

1. **新手能看懂**: 文档本身就是学习材料
