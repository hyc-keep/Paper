# 模板2: 设计依据(01_设计依据.md)

> **使用场景**: 每个需要"为什么这样设计"解释的阶段,比如05_LKMA、06_Boundary、07_Distance等
> **调用顺序**: 先看 `00_技能体系导航.md`,再在 `模板1_阶段总协议.md` 已冻结阶段边界后使用
> **与中央门禁的关系**: 本模板负责回答“为什么这样设计”; 但是否允许真正进入实现、是否已完成前后留痕,仍以中央 `crc-gland-coding-guard` 与 `模板5~10` 为准

---

## 单文档双达标模式（按需启用）

默认情况下,这份设计依据首先负责把“为什么这样设计”正式冻结下来。

只有当用户明确要求“这份 `01_设计依据.md` 本身也要兼顾教学解释,让新手直接读懂取舍逻辑”时,才升级为**单文档双达标模式**。

进入该模式后,除了正常回答设计问题、冻结边界和验收口径,还必须额外补齐:

- 每个 `##` 主章节开头 3 行内先抛一句结论,告诉读者这章最重要的判断是什么
- 至少保留 1 组候选方案对比,把“为什么不用更复杂/更直觉的方案”说透
- 至少保留 1 组误区预防,提前纠正“看起来更强但其实越界”的误读
- 至少保留 1 组联读顺序,告诉读者设计依据和阶段协议、文献提取、代码落点该怎么串起来读
- 至少保留 1 组五分钟自检,让读者能自己复述“问题是什么、为什么选它、不选什么、怎么验收”

**硬边界:**
- 双达标模式不允许把本模板写成泛泛而谈的综述文
- 不能因为要讲得好懂,就删掉单变量原则、验收标准、代码约束和论文对应关系
- 如果这份设计依据开启了双达标模式,人工终审时要同时参考 `学习型说明文人工审稿清单.md` 的提问框架

---

```markdown
# [模块名称]设计依据

## 0. 先给结论（仅在单文档双达标模式启用时强制）

- 当前最值得先做的设计: [一句话]
- 这个设计最主要解决的残余问题: [一句话]
- 为什么现在不用看起来更强的方案: [一句话]
- 如果你只想最快读懂本文件: 先看 `## 3. 任务难点与设计动机` → 再看 `## 4. 设计边界` → 最后看 `## 5. 本阶段的验收标准`

本文件不是"[模块名称]可能有帮助"的直觉说明,也不是把几篇论文拼在一起的讨论稿。

它在当前项目中的唯一职责是:
> 把[为什么先做这个模块/这个模块真正要验证什么/为什么这样定义模块边界/为什么当前只做最小验证]一次写死;
> 后续同层的[02_规则.md/03_步骤.md/04_保留标准.md]只能在这里冻结的设计前提下展开。

## 1. 文件角色与执行边界

### 1.1 当前文件负责什么

**只负责回答这6个设计问题:**

1. **为什么[本模块]在这个位置?** (必须在[上一阶段]之后,[下一阶段]之前)
2. **[本模块]到底要解决什么问题?** (残余误差类型/任务难点)
3. **为什么这个问题指向[当前技术方向]?** (文献支撑)
4. **为什么不直接用[更复杂的方案]?** (最小验证原则)
5. **这个设计的边界在哪里?** (做什么/不做什么)
6. **如何判断这个设计是否有效?** (验收指标选择)

### 1.2 当前文件不负责什么

**明确不在这里定义的内容:**

| 不负责的内容 | 以哪个文件为准 |
|------------|--------------|
| 具体实现参数(位置/kernel size等) | `[阶段]/02_[规则文件].md` |
| 详细的run执行顺序 | `[阶段]/03_实验步骤.md` |
| 最终保留/删除的裁决 | `[阶段]/04_保留或删除标准.md` |
| 上一阶段的设计 | `[上一阶段]/01_设计依据.md` |
| 下一阶段的设计 | `[下一阶段]/01_设计依据.md` |

### 1.3 为什么必须独立存在

**如果没有这份设计依据文件,会发生:**

- 同层文件02/03/04会各自解释"为什么这样设计",产生矛盾
- 代码实现时不知道"做到什么程度就停"
- 保留/删除判断时缺少"判断依据是什么"
- 论文写Methods时说不清楚设计动机
```

```
### 4.3 单变量原则声明

**本阶段严格遵守单变量原则:**

唯一改变的变量: `[变量名]`
固定不变的一切:
- 数据: 同01_数据协议的冻结版本
- 训练协议: `train_proto_v1`(同baseline)
- 评估协议: `eval_proto_v1`(同baseline)
- 其他所有超参数: 见`00_总览与规范/02_参数冻结总表.md`

**为什么单变量原则这么重要:**
你现在可能在想:"顺手多改几个参数,结果不是更好吗?"

这样想就错了。原因是:
- 如果同时改了3个变量,结果变好了,你根本不知道是哪个变量起了作用
- 论文写作时,你没法说"我们提出的[方法X]带来了Y个点的提升"——因为你不知道
- 审稿人会问:"请做消融实验。" 这时候你才发现没有单变量的对照
```
