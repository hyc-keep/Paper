# 模板3A: 训练评估型阶段验收(阶段验收.md)

> **使用场景**: 当前阶段会新增正式 `run`、`metrics`、可视化或模块性能比较时使用
> **进入前先确认**: 已先看 `模板3_阶段验收.md`，并确认当前阶段属于训练评估型
> **典型阶段**: `02_UNet流程验证`、`03_UNet稳定性`、`04_Baseline`、`05~09` 的大多数模型验证阶段

---

## 单文档双达标模式（按需启用）

默认情况下,本模板首先负责“基于 run / metrics / 可视化证据做正式放行”。

只有当用户明确要求“这份 `阶段验收.md` 本身也要兼顾教学解释”时,才升级为**单文档双达标模式**。

进入该模式后,除了保留正式验收标准,还必须额外补齐:

- 开头先给读者一句结论: 这轮结果到底能不能继续交给下游
- 至少 1 组“为什么不是只看单个最好数字”的解释
- 至少 1 组“高频误区/误判提醒”,提前说明什么情况下最多只能写 `partial`
- 至少 1 组“联读顺序”,告诉读者该怎样从 `run` 结果读到阶段结论
- 至少 1 组“五分钟复核动作”,让读者能自己打开 `val_metrics.csv / test*.csv / run_meta.yaml` 复验

**硬边界:**
- 双达标模式是“加解释层”,不是把训练评估型阶段验收写成散文化总结
- 不能因为补教学解释,就删掉正式 run 资产、定量验收标准、红线条件和阶段通过声明
- 如果这份正式验收启用了双达标模式,人工终审时必须同时参考 `学习型说明文人工审稿清单.md` 的问题框架

## 1. 当前文件的角色边界

### 当前文件负责什么

- 负责给“会新增正式 `run / metrics / 可视化 / 稳定性比较`”的阶段提供正式验收正文模板
- 负责把训练评估型阶段最关键的放行问题写成可检查的物理标准，而不是主观总结
- 负责把结果证据回链到真实工程资产，例如 `experiments/[run_name]/`、`reports/tables/`、`reports/figures/`

### 当前文件不负责什么

- 不负责替代 `模板3_阶段验收.md` 的总入口裁决角色
- 不负责治理汇总型阶段的文档闭环、止损裁决和收官判断；这类任务应回到 `模板3B_治理汇总型阶段验收.md`
- 不负责替代 `Post-QC Guard`、`diagnostics_result.txt` 或阶段 `README.md` 的最终同步职责

## 2. 先判断当前阶段属于哪一类

满足下面特征时，继续使用本模板：

- 本轮会新增或重跑正式 `experiments/[run_name]/`
- 本轮会产生新的 `val_metrics.csv`、`testA_metrics.csv`、`testB_metrics.csv` 或 `test_metrics.csv`
- 本轮重点是比较 baseline、模块增益、多 seed 稳定性、可视化质量
- 本轮的核心问题是“这轮结果够不够支撑下游继续用”

如果更像下面这些情况，就不该继续硬套本模板，而应切到 `模板3B_治理汇总型阶段验收.md`：

- 主要工作是汇总、归档、裁决、收官，而不是再产出新 run
- 主要交付物是阅读入口、汇总表、结论文档、止损判断
- 核心问题变成“这些已有结果和文档够不够真、够不够全、够不够一致”

## 2. 定量验收标准

### 2.1 主指标阈值

> 训练评估型阶段不等于“只看数字高不高”。
> 尤其像 `02_UNet流程验证` 这类首轮闭环阶段，更稳的通过线应该写成“协议是否跑通、闭环是否可信”，而不是一上来就把验收简化成单个指标阈值。
> 如果当前阶段本质上是“流程验证 / 协议冻结 / 首版闭环”，允许把主指标表写成“闭环门控 + 辅助数值”，不要硬伪装成“最终性能门槛”。

| 指标 | 当前阶段标准 | 当前实际值 | 是否通过 |
|------|-------------|----------|---------|
| [主指标1] | [阈值或方向] | [值] | [pass/partial/fail/not_applicable] |
| [主指标2] | [阈值或方向] | [值] | [pass/partial/fail/not_applicable] |

**为什么是这个阈值：**
- [写清楚阈值来自 baseline、文献值、路线锁定还是当前阶段“跑通”要求]

**如何核对实际值：**
1. 打开 `experiments/[run_name]/val_metrics.csv`
2. 如有正式测试结果，再打开 `testA_metrics.csv` / `testB_metrics.csv` / `test_metrics.csv`
3. 对照主指标列逐项核查

### 2.2 协议一致性检查

| 检查项 | 检查方法 | 通过标准 |
|------|---------|---------|
| 版本链一致 | 对比 `run_meta.yaml` | `config_version / data_proto_version / train_proto_version / aug_version / eval_proto_version / model_version / loss_version / postprocess_version` 完整且一致 |
| 命名规范一致 | 对比 run 目录与结果文件名 | 符合 `03_命名与结果记录规范.md` |
| 评估口径一致 | 核对 `best_selector` / `threshold_source` | 与 `04_评估口径与官方脚本对齐.md` 一致 |

### 2.3 稳定性要求（如果适用）

| 要求 | 检验方法 | 通过标准 |
|------|---------|---------|
| 多 seed 稳定性 | 固定 `3407 / 1234 / 2025` 三个 seed 比较 `mean +- std` | 先确认版本链一致，再判断波动是否在当前阶段噪声线内 |
| 结果方向一致 | 查看三次结果方向 | 不能出现“mean 为正但大多数 seed 反向” |
| 增益强度（模块阶段） | 计算 `delta_vs_baseline` 与 `std` 的关系 | 提升幅度应明显高于噪声线；更稳的经验口径是 `delta > 1.5 * std` |

**红线提醒：**
- 只有 1 次最好结果，不能写“稳定”
- 协议偷偷变化后得出的低 `std`，不能当稳定证据
- 提升量和噪声线同量级时，最多写 `partial`

### 3.1 可视化检查

| 样本类型 | 在哪里找 | 检查什么 | 通过标准 |
|---------|---------|---------|---------|

### 3.2 论文可写性检查

| 检查项 | 标准 | 当前状态 |
|-------|------|---------|
| 结果能支撑论文声明 | 数字提升或结论方向足够明确 | [pass/partial/fail/not_applicable] |
| 设计有文献支撑 | 设计依据与文献提取文档可回链 | [pass/partial/fail/not_applicable] |
| 有单变量证据 | 能说明结果变化来自当前阶段允许变量 | [pass/partial/fail/not_applicable] |
| 有展示资产 | 可视化质量足以进入报告或论文图池 | [pass/partial/fail/not_applicable] |

## 4. 红线条件

| 红线 | 如何检测 | 触发后的操作 |
|------|---------|-----------|
| 测试集泄露 | 检查 split 记录与样本来源 | 回退到数据阶段重查 |
| 结果不可复现 | 重跑后差异超出可接受范围 | 重新检查 seed 和协议字段 |
| 评估口径不一致 | `best_selector` / `threshold_source` / 指标文件命名不符 | 回退到评估协议检查 |
| 系统性预测偏移 | 可视化出现明显空间错位 | 回退到数据读取/预处理链 |
