# 模板7: 当前代码库状态

> **使用场景**: 正式动手前扫描工程现状,判断已有实现、空壳文件、历史资产和风险点。
> **调用顺序**: 先看 `00_技能体系导航.md`,并确保 `模板5` 与 `模板6` 已完成
> **与中央门禁的关系**: 本模板负责回答“工程里现在到底已经有什么”; 它不替代 Stage Gate,但会给 Stage Gate 和 Post-QC 提供现实参照

---

这份模板不能写成一句“代码库已扫描”。

真正关键的是:
- 哪些目录本轮必须扫到
- 哪些文件只是占位壳子
- 哪些历史 run / 报告 / external 适配层会被本轮改动影响
- 哪些风险是“现在不看,后面一定炸”
- 每个必扫目录至少要留下 1 个**真实样本路径**，而不是只写“目录已存在”
- 每个必扫目录至少要留下 1 个**量化信号**（文件数 / 样本数 / 字段值 / 行数），否则不算真实扫描

这份扫描不是凭感觉做的,最小依据至少来自:
- `模板5_Pre-check提取.md` 中已经提取出的本轮约束
- `模板6_阶段门控检查.md` 中已经确认的允许范围
- `00_总览与规范/01_工程目录框架.md`
- `00_总览与规范/05_代码工程映射与实现策略.md`

如果这四类依据都没对上,这里写出来的“当前代码库状态”大概率只是表面观察,不是正式工程判断。

```markdown

| `configs/` | 确认配置体系和版本字段 | 看是否已有相近配置可复用 |
| `src/` | 确认已有模块、接口和占位壳 | 看是否已有同类实现 |
| `scripts/` | 确认训练/评估入口脚本 | 看入口是否已经存在 |
| `tools/` | 确认辅助工具链 | 看是否已有检查脚本和生成脚本 |
| `b_class_auxiliary/` | 确认 B 类门禁脚本、运行留痕和治理产物是否已经按新结构隔离 | 看 `tools/`、`runtime_checks/`、`coding_guards/`、`workflow_gates/` 是否都已落位 |

**填写红线:**
- `已有文件/资产` 不能只写“正式数据目录 / 项目源码模块 / 历史 run 资产”这种抽象名词，至少要带 1 个真实文件路径锚点
- `状态` 不能只写“已存在 / 已实现”，最好同时带 1 个量化信号，例如“当前识别到 7 个 CSV”“当前已有 52 个源码文件”“当前已有 1 个正式 run 说明文件”
- 如果目录当前确实为空，也要诚实写成“空目录 / 暂无正式资产 / 当前不适用”，不要假装已经扫出了内容

## 5. 预期工程落点汇总

|------|----------|------------------|
| 代码层 | `src/[模块]/[文件].py` / `scripts/[脚本].py` / `tools/[脚本].py` / `b_class_auxiliary/tools/[脚本].py` | [本轮要新增或修改什么能力] |

**这一节的目的:**
- 把“扫描到了什么”进一步收敛成“真正会碰到哪些工程对象”
- 给后面的 `Pre-check Guard`、`Post-QC Guard` 和 diagnostics 留一个可对照的清单

**红线提醒:**
- 如果你没有扫 `experiments/` 和 `reports/`,就不要声称自己掌握了“当前已有资产”
- 如果你没有扫 `external/`,就不要声称“本轮不影响第三方适配层”
- 如果发现同类实现已经存在,这里必须写出来,否则后面很容易重复造轮子
- 如果 `datasets/`、`splits/`、`configs/`、`src/`、`scripts/`、`tools/`、`b_class_auxiliary/`、`experiments/`、`external/`、`reports/` 这些目录里真实有文件,却没有留下任何样本路径锚点,`check_precheck_docs.py` 会直接把这份扫描判成不合格
```
