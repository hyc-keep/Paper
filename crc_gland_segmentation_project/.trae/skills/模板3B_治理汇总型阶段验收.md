# 模板3B: 治理汇总型阶段验收(阶段验收.md)

> **使用场景**: 当前阶段主要任务是治理、汇总、裁决、归档、止损判断，而不是再产出一轮新模型结果
> **进入前先确认**: 已先看 `模板3_阶段验收.md`，并确认当前阶段属于治理汇总型
> **典型阶段**: `01_数据协议` 的最终放行收口、`10_结果汇总`、`11_总验收与止损`

---

## 单文档双达标模式（按需启用）

默认情况下,本模板首先负责“治理、汇总、裁决、收官”的正式验收。

只有当用户明确要求“这份 `阶段验收.md` 本身也要兼顾教学解释”时,才升级为**单文档双达标模式**。

进入该模式后,除了保留正式治理裁决,还必须额外补齐:

- 开头先给一句结论: 现在到底能不能继续、能不能收官
- 至少 1 组“为什么不能只看 README 口语化结论”的解释
- 至少 1 组“高频误区/误判提醒”,例如把 `diagnostics=partial` 误写成阶段完全 pass
- 至少 1 组“联读顺序”,告诉读者如何从汇总表、阶段入口、guard 回链到正式结论
- 至少 1 组“五分钟复核动作”,让读者自己查验总表数字、状态口径和 guard 是否一致

**硬边界:**
- 双达标模式是“补阅读引导”,不是把治理汇总型阶段验收降级成 README 式口语摘要
- 不能因为强调小白友好,就删掉正式汇总表回链、状态口径检查和阶段通过声明
- 如果这份正式验收启用了双达标模式,人工终审时必须同时参考 `学习型说明文人工审稿清单.md` 的问题框架

## 1. 当前文件的角色边界

### 当前文件负责什么

- 负责给“治理、汇总、裁决、收官”类阶段提供正式验收正文模板
- 负责检查已有结果、已有文档、已有结论是否足够真实、足够完整、足够一致
- 负责把治理类结论回链到正式结果表、阶段入口文档和 guard 留痕，而不是只给一句抽象判断

### 当前文件不负责什么

- 不负责替代 `模板3_阶段验收.md` 的总入口与模板选型职责
- 不负责训练评估型阶段的 run 验收正文；如果本轮重点是新增结果、比较指标和稳定性，应回到 `模板3A_训练评估型阶段验收.md`
- 不负责替代 `Post-QC Guard`、`diagnostics_result.txt`、结果总表或 `implementation_tracking` 入口文档本身

---

## 2. 先判断当前阶段属于哪一类

满足下面特征时，继续使用本模板：

- 本轮主要任务是汇总已有结果，而不是再训练一个新模型
- 本轮重点是统一状态口径、校验总表、维护阅读入口、做止损或收官判断
- 本轮核心问题是“这些已有资产和说法到底够不够真、够不够全、够不够一致”
- 本轮交付物更偏向 `README.md`、`implementation_status.md`、`reports/tables/`、结论文档和 guard 回链

如果更像下面这些情况，就不该继续硬套本模板，而应切到 `模板3A_训练评估型阶段验收.md`：

- 本轮会新增或重跑正式 `experiments/[run_name]/`
- 本轮要比较新的 `val/test` 指标、模块增益或多 seed 稳定性
- 本轮的核心问题是“这轮训练评估结果够不够支撑下游继续用”

## 4. 正式工程落点与回链对象

写完这份治理汇总型阶段验收，至少要能回链下面这些真实工程对象：

| 对象 | 正式工程落点 | 为什么必须回链 |
|------|-------------|---------------|
| 阶段验收正文 | `结直肠腺体分割_plan_优化版/01_实验执行/[当前阶段]/阶段验收.md` | 这是当前阶段正式裁决文件本体 |
| 阶段入口文档 | `reports/stage_reports/implementation_tracking/[当前阶段]/README.md`、`implementation_status.md` | 这里承载读者真正看到的阶段状态与阅读顺序 |
| 汇总表与总图 | `reports/tables/[文件].csv`、`reports/figures/[文件].png` | 这里承载结论文案背后的可审计证据 |
| 审计留痕 | `b_class_auxiliary/coding_guards/[任务目录]/` | 这里承载本轮 `Pre-check / Post-QC / diagnostics` 的回溯链 |
| 上游结果来源 | `experiments/[run_name]/`、`run_meta.yaml`、`val_metrics.csv`、`test*.csv` | 这里承载总表数字和结论的原始来源 |

**最小要求:**

- 不能只写“项目现在可以收官/继续”，却说不清这个判断来自哪些表、哪些文档、哪些 guard
- 不能只写 README 或 implementation_status 的口语化结论，却不回链真实结果文件
- 不能只在本模板里写治理结论，却不与 `Post-QC Guard`、阶段入口文档和总表同步

## 2. 核心验收问题

### 2.1 结果是否真实可回链

| 检查项 | 检查方法 | 通过标准 |
|------|---------|---------|
| 结果表中的数值能回链到正式文件 | 从汇总表反查 `run_meta.yaml` / `val_metrics.csv` / `test*.csv` | 每个关键数字都能找到来源 |
| 结论文案没有夸大完成度 | 对照 `implementation_status.md` 与 guard 记录 | 文案与真实状态一致 |
| 阶段结论与 diagnostics 一致 | 对照 `diagnostics_result.txt` 与 `Post-QC Guard` | 不出现“diagnostics=partial，但阶段宣称完全 pass” |

### 2.2 文档是否闭环

| 检查项 | 检查方法 | 通过标准 |
|------|---------|---------|
| README 导航有效 | 按推荐顺序打开文档 | 读者不会迷路 |
| implementation_status 诚实 | 核对“已完成 / 未完成 / 阻塞项” | 不夸大、不偷换 |
| 学习型说明文覆盖足够 | 对照对象-说明文映射表 | 核心正式对象均有对应说明 |

### 2.3 是否足以支持下一步或收官

| 判断问题 | 当前答案 | 判定标准 |
|---------|---------|---------|
| 下游阶段还能不能继续 | [是/否/部分] | 前置资产是否齐备 |
| 当前项目是否已经形成可写结论 | [是/否/部分] | 核心结果和论证链是否完整 |
| 是否需要止损/暂停扩展 | [是/否/部分] | 提升是否不足、风险是否过高、证据是否不足 |

> **高频混淆提醒**
>
> `治理汇总型阶段验收结论` 和 `package_status` 不是一个东西。
>
> - 阶段验收结论仍然只允许写 `pass / partial / fail / not_applicable`
> - 如果当前阶段同时在维护结果包分级，结果包状态只允许写 `ready / ready_with_minor_gaps / not_ready`
> - 后者服务“写稿/结果包是否可交付”，前者服务“这个阶段整体是否已经收口”
>
> 不允许直接把 `ready_with_minor_gaps` 原样抄成阶段总状态。

## 3. 协议与口径一致性检查

### 3.1 状态口径

这里第三列检查的是“你有没有把状态空间用对”，所以填写的仍然是本条检查的结论，而不是把左边那套状态空间再抄一遍。

| 对象 | 应使用的状态空间 | 检查结果 | 检查说明 |
|------|----------------|---------|---------|
| Stage Gate | `allow / blocked` | [pass/partial/fail/not_applicable] | [是否误写成 `pass/fail`] |
| Diagnostics / Post-QC | `pass / partial / fail / not_applicable` | [pass/partial/fail/not_applicable] | [是否与 guard 留痕一致] |
| implementation_tracking 阶段入口 | `not_started / blocked / partial / pass` | [pass/partial/fail/not_applicable] | [是否把局部 diagnostics 偷换成阶段总体状态] |

### 3.2 路径与命名口径

| 检查项 | 检查方法 | 通过标准 |
|------|---------|---------|
| 结果文件命名一致 | 对照 `03_命名与结果记录规范.md` | 不出现别名或旧口径 |
| 指标文件身份一致 | 对照 `GlaS testA/testB` 或 `CRAG test_metrics` 口径 | 文件命名与阶段身份一致 |
| 汇总表字段一致 | 对照 `result_tag / aggregation` | 表格身份与源文件一致 |

## 5A. 单文档双达标附加块（仅在单文档双达标模式启用时保留）

### 5A.1 为什么不能只看 README 的一句话结论

- 原因1: README 负责阅读入口,不替代正式裁决
- 原因2: 总表数字必须能回链到 `run_meta.yaml` 与指标文件
- 原因3: guard 和 diagnostics 不一致时,口语化说法不能单独成立

### 5A.2 高频误判提醒

- 误判1: “README 说基本完成,阶段就等于 pass”
  - 正确理解: [纠正]
- 误判2: “结果包 ready_with_minor_gaps 可以直接抄成阶段 pass”
  - 正确理解: [纠正]

### 5A.3 联读顺序

1. 先看 `reports/tables/[文件].csv` 或正式汇总表,确认关键数字
2. 再看 `b_class_auxiliary/coding_guards/[任务目录]/diagnostics_result.txt` 与 `Post-QC Guard`,确认状态口径
3. 最后回到本文件 `## 5. 阶段通过声明`,确认为什么当前只能写成这个治理结论

### 5A.4 五分钟复核

- 随机抽 1 个关键数字,从总表反查到 `experiments/[run_name]/test*.csv`
- 对照 `reports/stage_reports/implementation_tracking/[阶段名]/implementation_status.md`,说清 `implementation_tracking` 阶段入口为什么不能直接使用 `pass / partial / fail / not_applicable`
- 核对 `README.md`、`implementation_status.md`、`Post-QC Guard` 三者是否对当前完成度说法一致,并指出出现冲突时应以哪个正式文件先纠偏
