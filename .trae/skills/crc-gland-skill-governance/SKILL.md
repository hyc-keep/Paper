# CRC Gland Skill Governance

## 目标

这个 skill 只负责一件事:

> 当你正在修改 `.trae/skills`、项目导航、模板总规范或规程时,把体系做减法,直到它不再互相打架。

它是维护 skill 体系的管理 skill,不是日常实验入口。

## 什么时候调用

只有下面这些情况才调用:

- 删除或裁剪 skill
- 重写导航、模板总规范或规程
- 判断当前规则体系是否已经稳定

如果不是在改规则体系本身,不要调用它。

## 治理完成的最低标准

1. 根目录只有一个总入口: `crc-gland-coding-guard`
2. 项目内只有一个本地导航: `00_技能体系导航.md`
3. 模板总规范只有一份: `论文实验项目文档编写规范_完整版.md`
4. 学习型说明文规程只有一份: `实验执行文档编写规范.md`
5. 示例和历史稿不能再冒充正式依据
6. `skill 路由`、`非 skill 门禁`、`任务总放行件` 三层已经分清
7. 固定顺序已经写死: `治理 -> 研究 -> 阶段锁定 -> Pre-check -> 编码 -> 真实运行证据 -> 代码质量门禁 -> 总放行 -> 文档 -> 交付前清理`
8. 每次开始或继续任务时,都先显式展示“本轮已遍历的 skill / 导航 / 规程 / 当前阶段正式文件”清单,再给阶段判断
9. 当前仍在治理,或当前编号阶段仍未正式收口时,禁止在结论里顺嘴提下一编号阶段
10. 学习型说明文相关规程已经明确写死“先列交付范围内正式对象清单,再做逐文件映射”,不能再允许只写阶段入口页就声称覆盖代码说明
11. 文件三分归属已在 `制度完成定义.md` 写死: 每个文件只能属于 A(项目正式对象) / B(AI 辅助与流程文件) / C(存疑待清理) 之一,禁止混合件

## 交付前清理（固定顺序最后一步）

“交付前清理”是整条固定顺序的收尾,只在准备把项目打包交付给别人时执行:

1. 对全项目做一次 A / B / C 三分核对(判定口径以 `制度完成定义.md`「文件三分归属」为准)
2. A 类(项目正式对象): 保留,确认每个都有逐文件说明文
3. B 类(AI 辅助与流程文件): 确认可从交付包移除,不影响项目产物链运行
4. C 类(存疑待清理): 逐个裁决,转入 A(补依据与说明文)或 B(移除)
5. 重点排查“混合件”: 任何把项目内容和 AI 辅助检查内容写在同一文件里的情况,必须先拆成纯 A 和纯 B 两个文件,再决定去留
6. 清理结论必须留痕,写清哪些文件归 A 保留、哪些归 B 移除、C 类如何裁决及理由

## 本 skill 的产物

完成后至少要能回答:

- 现在唯一入口是什么
- 现在哪些文件只是被动模板
- 当前是否允许进入研究定标: `allow / blocked`
- 当前每一轮路由判断是否留下了显式已读清单
- 当前是否已经把“编号阶段下一步”和“本轮工作流下一步”拆开表达

## 红线

- 规则体系还没收敛,就放行研究或编码
- 为了解决混乱,继续新增并列入口
- 把模板、示例、规程写成新的总路由
- 没有先展示本轮已遍历清单,就直接宣称“当前该去哪个阶段”
- 当前仍在治理或当前编号阶段仍未收口,却提前提议下一编号阶段
- 规则里还允许把 `README.md` / `implementation_status.md` 当成真实工程对象逐文件说明的替代品
