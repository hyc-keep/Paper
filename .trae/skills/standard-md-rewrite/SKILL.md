# 标准实验文档重写协议

## 目的

这个 skill 用于强制执行本项目实验执行文档的严格重写协议。

它的存在目的，是防止把项目规范当成“读过一遍就算完成”的参考清单。目标不是只读 `00_总览与规范`，而是在正式判定某个目标文件完成之前，真正提取、应用并核验其中的规则。

## 在 skill 体系中的定位

本 skill 只负责 `结直肠腺体分割_plan_优化版/01_实验执行` 下 plan 正文的重写与质检，属于计划文档重写工具。

它不是代码工作流的总入口。代码工作流的唯一路由入口是 `crc-gland-coding-guard`。本 skill 不参与 `治理 -> 研究 -> 阶段锁定 -> Pre-check -> 编码 -> 运行证据 -> 代码质量 -> 说明文档` 这条链的路由判断，也不替代任何 gate report。

它不是“高优先级建议”，而是硬性执行协议。命中适用范围后，不允许再以“我理解差不多”“正文已经够完整”“这次先简写一下”为理由自行放宽。

## 何时调用

当出现以下任一情况时，必须调用本 skill：

- 用户要求编辑、重写、审查、修补、标准化或质检 `结直肠腺体分割_plan_优化版/01_实验执行` 下的任意 `md`
- 用户要求判断某个阶段目录或某个 `md` 是否“达到标准”
- 用户要求继续推进 `01_实验执行` 下任意阶段的工作


不要把本 skill 当成可选建议。只要任务命中上述范围，它就是必须遵守的执行协议。

如果已经调用本 skill，后续执行中仍出现与本 skill 明文要求不一致的做法，必须优先判定为执行违规，而不是优先相信操作者的主观判断。

## 适用范围

- 项目范围：`结直肠腺体分割_plan_优化版/01_实验执行`
- 当前冻结阶段顺序：
  - `01_数据协议`
  - `02_UNet流程验证`
  - `03_UNet稳定性`
  - `04_Baseline`
  - `05_LKMA`
  - `06_Boundary`
  - `07_Distance`
  - `08_外部对比`
  - `09_CRAG验证`
  - `10_结果汇总`
  - `11_总验收与止损`
- 不允许跳阶段。如果更早阶段还没有按协议真正完成，不得进入后续阶段。

## 核心原则

首要任务永远是目标 markdown 正文本身。

清单与自检段落都只是“已经正确重写”的辅助闭环，绝不能替代正式 markdown 正文的重写。

只要本 skill 对某一部分提出了“必须”“不得”“不能”“只允许”这类约束，就应按不可裁剪、不可缩写、不可弱化的硬规则执行，除非用户明确要求修改 skill 本身。

凡是本 skill 明确要求“写明”“补齐”“核对”“留痕”的内容，都必须以可回查的显式文本出现；不允许把它们藏在概括句、总结句、勾选句或“已经覆盖”式表述里冒充完成。

如果某项内容在最终文档或最终汇报中不能被直接指出具体落点，就按“未写”“未补齐”“未核对”处理，而不是按“应该已经包含”处理。

下列情况都不能算通过：

- 只是重新读了一遍 `00_总览与规范`
- 只是补了一个质量自检
- 只是往原文后面追加了一些内容
- 只是因为“正文看起来已经比较完整”就自行压缩 `文件质量自检`、`代码落地接口`、`冲突裁决记录` 或 diagnostics 说明
- 只是凭主观判断，把本应逐项留痕的审计步骤改写成摘要结论
- 正文引用了某个上游文件，但“本轮重写直接依赖的前置文件”或同类留痕位置没有同步记录
- 在发现缺口后，没有先修正目标文件和闭环证据，就继续推进下一个文件

只有当正式 markdown 正文已经被真正重写到符合项目协议，并且必要闭环也完成时，文件才算完成。

## 零裁量约束

以下内容一律按零裁量执行，不允许代理自行解释为“可简化”：

- `00_总览与规范` 的完整重读
- `02_路线与投稿`、`03_文献证据` 与强相关前后文的继续阅读
- 正文中的路线依据、文献依据、门控链、验收与回退写明
- 正文中的 `本轮重写直接依赖的前置文件` 或同等作用的独立留痕段
- 正文中的独立 `回退条件` 段；不能只在验收、放行或一句总述里顺带提到
- `代码落地接口`
- `冲突裁决记录`
- `文件质量自检`
- diagnostics 闭环
- “已读文件 / 正文落点 / 自检落点 / diagnostics” 之间的一致性核对
- 用户发起质疑或审计时的逐项对表留痕
- 同批或同层已完成文件的 `文件质量自检` / `Diagnostics 闭环` 模板强度对照
- 如果当前文件使用了比同批文件更弱的收尾模板，则必须先补齐到同等强度，才能宣称完成

上述任一项缺失、缩水、漏记或只做口头说明，都不得视为“基本完成”。

## 强制工作流

对每一个目标 markdown 文件，都必须按以下顺序执行。

### 第 1 步：在处理任意目标 `md` 前重新完整遍历 `00_总览与规范`

在编辑、重写、审查、质检 `01_实验执行` 下任意 markdown 之前，必须重新读取并实际应用以下全部文件：

- `00_总览与规范/00_执行导航.md`
- `00_总览与规范/01_工程目录框架.md`
- `00_总览与规范/02_参数冻结总表.md`
- `00_总览与规范/03_命名与结果记录规范.md`
- `00_总览与规范/04_评估口径与官方脚本对齐.md`
- `00_总览与规范/05_代码工程映射与实现策略.md`
- `00_总览与规范/06_实验执行证据化写作模板.md`
- `00_总览与规范/07_实验执行全局修订与质检规范.md`

重要要求：

- 不能只读就算完成
- 必须提取这些文件中适用于当前目标文件的规则
- 必须把这些规则真正应用到当前目标文件
- 必须在最终正文、自检或审计结论中留下可回查的落点，不能只在脑中使用后不留痕

### 第 2 步：继续完成 `00_总览与规范` 之外的上游阅读

读完 `00_总览与规范` 后，必须继续遍历所有相关上游文件：

- `02_路线与投稿` 下的相关文件
- `03_文献证据` 下的相关文件
- `01_实验执行` 中已经写过且与当前目标文件强相关的前后文文件

不能停在 `00_总览与规范`。项目规范明确要求继续完成上游与上下文阅读。

如果 `00_总览与规范` 进一步规定了当前任务必须继续执行的动作，例如：

- 文献应该怎么读、怎么提取
- 正文应该怎么组织和怎么写
- 前后文应该怎么互校
- 质检应该怎么做、哪些情况不得通过

那么这些都不是“已阅读即可”的参考说明，而是必须继续执行到位的后续步骤。只完成文件遍历、不继续落实这些步骤，不能算通过。

如果正文后面实际引用了某个上游文件、同层文件或下游文件，那么该文件必须同步出现在本轮阅读留痕或角色说明中；不允许出现“正文已使用，留痕未同步”的证据链缺口。

### 第 3 步：按文件角色做差异化遍历

禁止使用统一的“邻近文件列表模板”。

针对当前目标文件，必须明确识别并说明：

- 上游文件
- 同层文件
- 下游文件

这些文件的选择必须由当前 markdown 文件的真实角色决定。

例如：

- 阶段总协议文件，必须广泛检查同层细则，并检查下游阶段交接要求
- 数据源类文件，必须突出数据身份、split 消费者、资产交接和下游 benchmark 使用方式
- 数据检查类文件，必须突出门控逻辑、通过/不通过条件，以及下游训练入口阻断关系
- 结果或验收类文件，必须突出交接条件、回退触发条件和后续阶段消费方式

在最终正文与说明中，必须清楚写出“为什么这些文件对当前文件相关”，而不能只是列文件名。

禁止为了省事复用上一个文件的邻近文件模板。只要当前文件角色不同，就必须重做角色判断和相关性说明。

### 第 4 步：先提取，再重写

在修改目标 markdown 之前，必须显式提取：

- 路线依据
- 文献依据
- 定义、公式或原理
- 参数边界
- 代码落点
- 运行记录字段
- 验收规则
- 回退触发条件

禁止基于模糊印象直接重写。

如果提取内容没有真实落到正文、规则卡片、接口说明、自检或审计说明中，则视为“未完成提取应用”，不能算通过。

特别要求：

- `回退触发条件` 不能只在脑中提取后不写
- 如果自检声称“已写清回退条件 / 回退边界 / 回退要求”，那么正文里必须存在对应独立落点
- 如果自检声称“已写清某类依据或规则”，则必须能指出正文中的具体段落或标题；不能只靠“整体已经覆盖”

### 第 5 步：先重写正式 markdown 正文

正式目标 markdown 文件必须优先完成正文重写。

重写后的正文，必须足以直接支撑：

- 代码实现
- 实验执行
- 论文写作
- 阶段验收
- 回退

避免以下弱修改方式：

- 只追加几条说明
- 只补一个自检段落
- 只加几条参考文献
- 在旧结构已经不符合标准时，仍基本保持原结构不动

如果旧结构已经不能承载当前协议要求，应优先整文件重写，而不是局部打补丁后继续沿用旧结构。

### 第 6 步：补齐必需的收尾部分

每个正式目标 markdown 文件在适用时都必须包含以下部分：

- `代码落地接口`
- `冲突裁决记录`
- `文件质量自检`
- diagnostics 闭环

如果文件涉及规则、方法、loss、target、metric、postprocess 或阶段门控，还必须具备足够深度，能够直接支撑实现与写作。

其中，`文件质量自检` 不允许代理自行压缩为“摘要版”“简版”“代表性条目版”；除非用户明确要求缩写，否则必须按同批文件已形成的完整模板强度执行。

这里的“同批文件已形成的完整模板强度”不是抽象概念，而是必须实际挑选最近、最相似、已完成重写的同层文件作为对照模板，并显式核对：

- `文件质量自检` 条目颗粒度是否明显缩短
- `文件质量自检` 是否只保留少数代表性勾选项
- `Diagnostics 闭环` 是否仍保留独立标题与独立结论
- 当前文件是否删掉了对照模板中本应继续适用的关键检查项

只要上述任一问题存在，就按“收尾模板缩水”处理，不得标记完成。

diagnostics 也不能只写“已检查”。必须实际运行，并在最终说明中明确是否有问题、是否已修复。

如果同批文件已经形成显式标题 `Diagnostics 闭环`，当前文件不得退回到更弱的 `diagnostics`、`诊断`、`检查结果` 等模糊标题，除非用户明确要求统一改名。

如果文件涉及阶段门控、交接放行、验收失败、回退重跑、回流修订或下游消费条件，还必须补：

- 独立 `回退条件` 段

如果用户对执行质量提出质疑、要求审计、怀疑还有偷工减料，或本轮任务本身就是“审查/质检/核对是否按 skill 执行”，还必须补：

- 独立 `审计对表` 或同等作用的逐项对表段

上述两项不能被合并进一句总结，也不能用“已在正文体现”“已在自检覆盖”替代。

### 第 6.1 步：强制执行同批模板对照

在补齐 `代码落地接口`、`冲突裁决记录`、`文件质量自检`、`Diagnostics 闭环` 之后，必须额外执行一次“同批模板对照”：

- 选择最近一个同层、同阶段、已经按完整协议重写过的文件作为对照模板
- 对照 `文件质量自检` 的条目数量、条目颗粒度、覆盖维度、是否包含回退与 diagnostics 对齐说明
- 对照 `Diagnostics 闭环` 的标题、内容颗粒度与闭环结论写法
- 如果当前文件比对照模板更弱，必须先补齐，不得以“正文已经更强”作为豁免理由

这个步骤不是建议项，而是最终宣称完成前的必做闸门。


## 正文深度要求

正式 markdown 文件不能停留在表层说明。

适用时，必须清楚写出：

- 路线依据
- 文献依据
- 原理或公式
- 参数边界
- 代码落点
- 运行记录字段
- 验收方式
- 回退条件

标准是：文件必须强到足以直接翻译成代码和论文文字，而不需要靠猜。

如果仍然需要靠操作者补脑、靠下游文件二次猜测、靠口头说明兜底，则该文件不能判为合格。

### 显式落点要求

下列内容默认要求具有独立标题、独立小节或至少独立成段的清晰落点，不能仅散落在多处句子里让读者自己拼：

- `本轮重写直接依赖的前置文件`
- `路线依据`
- `文献依据`
- `代码落地接口`
- `冲突裁决记录`
- `文件质量自检`
- `Diagnostics 闭环`

对于涉及门控或交接的文件，还必须有：

- `回退条件`

对于审计任务，还必须有：

- `审计对表`

如果缺少这些显式落点，即使内容似乎“在别处提过”，也按未完成处理。

## 规则卡片标准

当文件包含正式规则卡片时，必须使用项目要求的字段：

- `当前结论`
- `规则类型`
- `适用阶段`
- `直接依据`
- `采用原因`
- `不采用的相邻方案`
- `代码落点`
- `运行记录字段`
- `验收方式`

如果规则涉及 loss、metric、target 或 postprocess，还必须补：

- `核心公式或定义参考`

## 按文件类型的遍历指引

必须根据当前文件类型决定遍历重点。

### A 类：总览 / 全局协议类文件

重点关注：

- 上游冻结规则
- 较广的同层覆盖
- 下游阶段交接关系

### B 类：数据协议类文件

重点关注：

- 数据身份
- split 依据
- 标签规则
- 检查与预览
- 正式资产
- 阶段验收
- 模型阶段如何消费这些结果

### C 类：模型 / 训练 / 模块类文件

重点关注：

- 继承的数据与评估协议
- 结构与 loss 设计
- 参数冻结
- 评估接口
- 与前序模型阶段的对比关系

### D 类：评估 / 结果 / 验收类文件

重点关注：

- 验收阈值
- 回退触发条件
- 证据资产
- 下游汇总与论文表格使用方式

## 通过闸门

只有同时满足以下条件，目标文件才允许判定为通过：

- 正式 markdown 正文已经被真正重写
- 当前文件所需的上游阅读已经实际完成
- `00_总览与规范` 中适用于当前任务的后续要求已经被真正执行，而不只是读取
- 已记录差异化的上游 / 同层 / 下游遍历
- 已写明真实提取内容
- 已补齐必需的收尾部分
- 已执行 diagnostics
- 本次修改未引入新的未解决 diagnostics 问题
- 已核对正文引用、前置阅读留痕、自检表述与 diagnostics 结论之间没有互相打架
- 已完成与同层最近合格文件的 `文件质量自检 / Diagnostics 闭环` 强度对照，且当前文件没有缩水
- 如果文件适用 `回退条件`，则正文已存在独立回退落点，自检不得只写“已写清回退边界”而正文无对应位置
- 如果本次属于审计/质检任务，则已产出 `已读文件 -> 正文落点 -> 自检落点 -> diagnostics -> 缺口/修复状态` 的逐项对表
- 若审计中发现缺口，已经先回写修正并重新复核，而不是带缺口继续推进后续文件

## 红线

只要出现以下任一情况，就不得把文件标为完成：


- 没有真实执行协议，只是把自检打勾
- 在 `00_总览与规范` 中只读了 `00_执行导航.md`
- 读了 `00_总览与规范`，但没有真正应用到当前文件
- 把 `00_总览与规范` 当成完成式阅读，没有继续执行其中指向的写作、文献、互校或质检步骤
- 上游阅读在 `02_路线与投稿` 和 `03_文献证据` 之前就停止
- 当前文件仍然不足以直接指导实现、执行、写作、验收和回退
- 代理自行把完整模板压缩成“自己觉得够用”的版本
- 审计已经发现留痕缺口、正文缺口或 diagnostics 问题，但没有先回写修复
- 使用“应该差不多”“大体一致”“正文已经很强”替代逐项核对和正式留痕
- 自检声称“已写清回退条件 / 已写清回退边界”，但正文没有独立 `回退条件` 落点
- 自检声称“已完成审计或逐项核对”，但没有实际产出 `已读文件 -> 正文落点 -> 自检落点 -> diagnostics -> 缺口` 对表
- 当前文件的 `文件质量自检` 明显弱于同批同层最近合格文件，却仍宣称“已按 skill 完成”
- 当前文件把 `Diagnostics 闭环` 缩成更弱标题或更弱内容，却没有用户明确授权
- 用结论式勾选、摘要式总结或“整体已覆盖”替代证据落点说明
- 最终汇报宣称“已按 skill 完成”，但说不出每类证据在正文中的具体落点

## 审计优先规则

如果用户对某个文件提出“不相信是否完全按 skill 执行”“怀疑还有偷工减料”“要求逐项审计”之类的质疑，后续任务优先级必须立即切换为审计，而不是继续推进下一个文件。

此时必须至少完成以下动作：

- 重新调用本 skill
- 重新重读 `00_总览与规范` 与最相关上游 / 同层 / 下游文件
- 对目标文件做“已读文件 -> 正文落点 -> 自检落点 -> diagnostics -> 缺口”的逐项对表
- 只要发现缺口，先修复目标文件并复跑 diagnostics，再决定是否放行

这个“逐项对表”不是脑内检查，也不是一句“已经对齐”。它必须以显式文本出现，至少覆盖以下列：

- 已读文件
- 该文件在正文中的落点
- 该证据在自检中的落点
- diagnostics 状态
- 当前缺口
- 修复状态

如果本次审计还涉及“为什么这篇和另一篇模板不一样”“是不是某篇自检缩水”之类的对比质疑，则逐项对表必须额外补两列：

- 对照模板文件
- 当前文件相对对照模板的缩水项 / 补齐项

如果某一列说不出来或没有落点，就按该项未完成处理。

在上述审计闭环完成前，不得擅自继续下一个文件。

## 最终汇报要求

在汇报完成情况时，必须清楚区分：

- 正式 markdown 正文到底真实重写了什么
- 跑了哪些 diagnostics
- 还剩下哪些阻塞项，如果有的话

如果本次是审计任务，还必须明确说明：

- 审计覆盖了哪些已读文件
- 每类证据在正文中的落点在哪里
- 自检与 diagnostics 是否同步
- 是否完成同批模板强度对照；对照对象是哪一篇
- 发现了哪些缺口
- 哪些缺口已经修复，哪些还未修复

如果任何强制协议步骤缺失，绝不能暗示该文件已经完全合规。

禁止使用以下话术替代正式汇报：

- “已经基本按 skill 做了”
- “正文已经覆盖得差不多”
- “质检应该没问题”
- “大体一致”
- “主要都补齐了”

只有当显式落点、逐项对表和 diagnostics 闭环都真实存在时，才允许使用“完成”“通过”“已按 skill 执行”这类完成式表述。
