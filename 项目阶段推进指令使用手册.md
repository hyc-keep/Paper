# 项目阶段推进指令使用手册

## 1. 这份手册是干什么的

这份手册只负责回答下面 6 个问题：

1. 每次开始一个任务时，应该怎么给指令
2. 任务推进到中间阶段时，应该怎么继续下达指令
3. 一轮任务准备结束时，应该怎么要求我做收口
4. 你应该怎么告诉我当前环境，尤其是本地 `sd` 环境和后续 GPU 服务器
5. 什么时候应该租 GPU，什么时候先不要租
6. 我怎样才能正确识别你的意图，并按 skill 的固定执行链做事

这份手册不是新的总入口。

项目的唯一总入口仍然是：

- `d:\12_Medical_Image_Segmentation\Paper\.trae\skills\crc-gland-coding-guard\SKILL.md`

固定执行链仍然只有一条：

- `治理 -> 研究 -> 阶段锁定 -> Pre-check -> 编码 -> 运行证据 -> 代码质量 -> 总放行 -> 说明文档 -> 交付前清理`

这里必须额外记住一个容易混淆但不能混的事实：

- `01_数据协议 / 02_UNet流程验证 / ...` 是编号阶段
- `治理 / 研究 / 阶段锁定 / Pre-check / 编码 / 运行证据 / 代码质量 / 总放行 / 说明文档 / 交付前清理` 是 skill 工作流阶段

编号阶段允许进入下一阶段，不等于当前 turn 可以跳过 `运行证据 -> 代码质量`。

## 2. 你需要先记住的原则

### 2.1 我会自动推进，但不会乱猜

只要你的指令给清楚，我会尽量自动完成当前阶段内部该做的事，包括：

- 读取 `结直肠腺体分割_plan_优化版`
- 读取 `结直肠腺体分割_正式参考资料`
- 判断当前应该停在执行链的哪一步
- 产出当前阶段必须留下的正式对象
- 在应该停下来的地方停下，不会擅自跳到后面

但这里的“后面”要分两层判断：

- 编号阶段的后面，例如 `01` 后面是 `02`
- skill 工作流的后面，例如 `编码` 后面必须先到 `运行证据`

如果本轮已经改了正式代码、配置或正式资产，我必须先补当前 turn 的 `运行证据 -> 代码质量`，再谈是否允许进入下一编号阶段。

但我不会自动猜下面这些事情：

- 当前到底在本地还是服务器执行
- 本轮是否允许动代码
- 本轮是否允许跑命令
- GPU 是否真的可用
- 服务器如何连接
- 当前 conda 环境叫什么
- 数据目录和工程目录这轮是否有变化

### 2.2 skill 负责制度，不负责猜机器

skill 应该负责：

- 阶段顺序
- 前置条件
- 必需产物
- gate 检查
- 禁止跳步

skill 不应该硬编码：

- 本地 `sd` 环境的具体路径
- 服务器 IP
- 服务器账号口令
- 哪一次任务一定在本地跑，哪一次一定在服务器跑
- 你临时租的 GPU 实例信息

这些是“执行上下文”，不是“制度本体”。

### 2.3 编码依据留痕不是说明文档阶段才补

这一点必须单独记住：

- `说明文档` 阶段负责把已经成立的实现讲清楚
- 但“我这轮写代码到底参考了什么、参数按哪份协议冻结、每个正式文件对应哪条依据”这件事，属于 `编码` 阶段就必须同步留下的正式留痕

这里的“每个正式文件”不是泛指模块，而是逐文件对象：

- 每个被改动的正式 `py` 文件都必须在 `实现依据记录.md` 里单独出现
- 如果本轮改了 5 个正式 `py` 文件，就至少要有 5 条逐文件依据回链
- 不是每个 `py` 文件各写一份 md，而是同一份 `实现依据记录.md` 对这些文件逐个审计
- 但这里的“同一份”只限当前编号阶段内部，不再允许跨阶段共用唯一一份根路径文件

也就是说：

- 研究阶段要回答“当前阶段为什么现在必须这样做”
- 编码阶段要回答“这份正式代码/配置/正式资产具体依据什么写出来”
- 说明文档阶段才负责把前两者和代码证据组织成通俗说明

不能等到说明文档阶段再回忆“我当时参考了哪些论文、官方脚本、参考代码、计划条款”。

从现在开始，`实现依据记录.md` 还必须满足下面这条归档规则：

- 每个编号阶段都必须在 `reports/stage_reports/implementation_tracking/[阶段名]/实现依据记录.md` 保留独立归档
- 根路径 `crc_gland_segmentation_project/b_class_auxiliary/runtime_checks/实现依据记录.md` 只允许作为“当前阶段副本”
- 进入新阶段编码前，禁止直接覆盖旧阶段唯一依据记录
- 如果要新开阶段，优先先运行 `crc_gland_segmentation_project/b_class_auxiliary/tools/init_stage_archive.py` 初始化阶段归档和根路径副本

### 2.4 每次都先展示本轮已遍历清单

如果你让我“继续”或“按 skill 判断现在到哪步了”,我必须先显式展示本轮已经遍历的:

- root skill
- 项目内导航 / 完成定义 / 手册
- 当前阶段正式文件
- 当前阶段最近一次 gate report

然后才能给阶段判断。

如果还没读齐,正确动作是先补读,不是先下结论。

### 2.5 不要把三层证据压成一层

从这轮开始,下面 3 层证据必须分开留:

1. `研究证据`
   - 解决“为什么现在做这个、依据什么锁边界”
   - 正式产物: `研究定标记录.md`
2. `实现依据证据`
   - 解决“这轮正式代码/配置/正式资产到底参考了什么写出来”
   - 正式产物:
     - `reports/stage_reports/implementation_tracking/[阶段名]/实现依据记录.md`
     - `b_class_auxiliary/runtime_checks/实现依据记录.md`（仅当前阶段副本）
3. `运行与质量证据`
   - 解决“代码到底真实跑到了哪里、质量是否成立”
   - 正式产物: `runtime_check_report.md`、`runtime_evidence.json`、`runtime_check.log`、`code_quality_gate_report.md`

这 3 层里,任何一层缺了都不能指望靠学习型说明文补回来。

这里尤其不要把“阶段归档版”和“根路径当前副本”混成一份滚动文件；阶段归档负责保留历史审计链，根路径副本只服务当前阶段门禁和查看入口。

## 4. 我怎样识别你的指令

我会优先从你的话里识别 4 类信息。

### 4.1 你要推进到哪一步

例如：

- “从 `01_数据协议` 的研究定标开始”
- “继续推进到阶段锁定”
- “当前只做 Pre-check，不进入编码”
- “当前只做 runtime evidence”
- “当前只做总放行，不写说明文”

### 4.2 你允许我做到什么程度

例如：

- “允许你读文档、改文件、跑命令”
- “只允许读和分析，不要改文件”
- “允许你写代码，但不要启动长时间训练”
- “只输出体检报告，不执行任何脚本”

### 4.3 当前环境是什么

例如：

- “当前环境是本地 conda `sd`”
- “当前只有 CPU，不要跑 GPU”
- “从训练开始切到服务器”
- “服务器环境叫 `seg`”

### 4.4 当前限制和偏好是什么

例如：

- “这轮不要写说明文”
- “这轮先只处理 GlaS，不碰 CRAG”
- “如果缺少信息先问我，不要猜”
- “先 smoke check，再决定是否正式训练”

## 5. 你每次下指令，最低要包含什么

以后每次发任务，尽量至少包含下面 4 句：

1. `按 skill 推进到哪个阶段`
2. `本轮允许我做到什么边界`
3. `当前环境是什么`
4. `如果缺信息，是先问你还是让我自己判断`

最小可用模板：

```text
按 skill 推进【阶段名】。
本轮只做到【边界】。
当前环境是【本地/服务器 + 环境名 + GPU情况】。
如果缺少必要信息，先问我，不要猜。
```

## 6. 开始阶段时怎么给指令

### 6.1 用途

用于：

- 从一个新阶段开始
- 从重启后的项目重新进入执行链
- 从 `治理` 正式切换到 `研究`

### 6.2 推荐写法

```text
按 skill 推进 `01_数据协议`。
本轮目标：先完成研究定标，不进入阶段锁定之后的步骤。
允许你读取 `结直肠腺体分割_plan_优化版` 和 `结直肠腺体分割_正式参考资料`，并在 `crc_gland_segmentation_project` 里产出正式研究记录。
当前环境：本地 conda `sd`，不要跑 GPU 任务。
如果缺少必要信息，先问我，不要猜。
```

### 6.3 我收到后会怎么做

我会按顺序做这些事：

1. 判断当前是不是还卡在治理阶段
2. 如果可以进入研究，就先读 `plan` 和 `正式参考资料`
3. 提取当前阶段硬约束
4. 产出 `研究定标记录.md`
5. 跑 `check_research_alignment_gate.py`
6. 报告是否 `pass`
7. 如果没有通过，只在当前阶段修补，不会偷偷跳到下一步

## 7. 中间推进时怎么给指令

### 7.0 先分清“编号阶段推进”和“当前 turn 收口”

如果你说“下一步进入 `02_UNet流程验证`”，我也必须先检查两件事：

1. 编号阶段门控是否真的允许从 `01` 放行到 `02`
2. 当前 turn 是否还欠 `运行证据 -> 代码质量`

只要第 2 条没完成，就不能把“可进入 `02`”说成“本轮已经完全收口”。

如果当前仍在 `治理`,或当前编号阶段的 formal artifact / gate 还没收口,就不要在这一步顺嘴提下一编号阶段。

### 7.1 研究做完后，推进到阶段锁定

```text
继续上一个任务，按 skill 判断当前是否允许进入阶段锁定。
如果 research gate 已通过，就起草 `00_阶段实现卡.md` 并运行阶段锁定 gate。
如果 research gate 没通过，就先补研究，不要跳步。
当前环境仍然是本地 conda `sd`。
```

### 7.2 阶段锁定后，推进到 Pre-check

```text
继续当前阶段，按 skill 推进到 Pre-check。
如果 `stage_definition_gate_report.md` 已经通过，就补齐 Pre-check 四件套并运行 precheck gate。
如果阶段锁定还没通过，就不要进入 Pre-check。
当前环境仍是本地 conda `sd`，不要开始正式训练。
```

### 7.3 Pre-check 后，推进到编码

```text
按 skill 推进到正式编码。
先检查 research / stage definition / precheck 是否都已通过。
只有前置 gate 全部通过后，才允许你修改正式代码和配置。
开始改动前，先为当前编号阶段建立阶段归档，并同步初始化根路径当前副本；优先运行 `b_class_auxiliary/tools/init_stage_archive.py --stage-name [当前阶段名]`。
然后再建立或更新 `实现依据记录.md`，把本轮计划依据、论文/文献依据、官方脚本/参考代码依据、冻结参数来源写清。
注意：当前阶段正式留痕应写入 `reports/stage_reports/implementation_tracking/[当前阶段名]/实现依据记录.md`，根路径 `b_class_auxiliary/runtime_checks/实现依据记录.md` 只作为“当前阶段副本”。
当前环境是本地 conda `sd`，这轮先只做代码实现和轻量 smoke check。
```

如果这一步没先做,后面即使代码写出来,也只能算“代码存在”,不能算“实现依据成立”。

如果这一步改了多个正式 `py` 文件,就必须在 `实现依据记录.md` 的“正式文件到依据的回链”表里逐文件填写,不能把多个文件压成一句“训练代码已按论文实现”。

如果当前阶段独立归档还不存在，或者根路径文件没有明确声明“当前阶段副本”，那就还不能进入正式编码。

### 7.4 编码后，推进到运行证据和代码质量

```text
继续当前任务，按 skill 推进到运行证据和代码质量门禁。
如果本轮确实改了正式代码、配置或正式资产，就先检查当前阶段 `实现依据记录.md` 独立归档和根路径当前副本是否都已经补齐，再生成 runtime 三件套并运行 code quality gate。
如果当前机器条件不适合正式训练，先告诉我阻塞点，不要硬跑。
当前环境是【请写清楚本地或服务器】。
```

### 7.5 这里的“证据”分三层,不要混

1. `研究证据`
   - 说明当前阶段为什么这样做
   - 产物是 `研究定标记录.md`
2. `实现依据证据`
   - 说明当前正式代码/配置/正式资产具体参考了什么
   - 产物是当前阶段独立归档 `reports/stage_reports/implementation_tracking/[阶段名]/实现依据记录.md`，以及根路径当前副本 `b_class_auxiliary/runtime_checks/实现依据记录.md`
3. `运行与质量证据`
   - 说明代码真的跑到了什么程度
   - 产物是 `runtime_check_report.md`、`runtime_evidence.json`、`runtime_check.log`、`code_quality_gate_report.md`

学习型说明文不是拿来补前面 3 层缺口的。

## 8. 结束阶段时怎么给指令

### 8.1 一轮任务准备收口

```text
请按 skill 检查当前任务是否已经允许进入总放行。
如果前置阶段有任何一个还没 pass，就不要口头说完成。
如果已经满足条件，就运行 workflow gate 并告诉我当前任务是 pass、partial 还是 blocked。
当前环境是【本地/服务器 + 环境名】。
```

这里的“满足条件”必须额外包含一条:

- 如果本轮改了正式代码、配置或正式资产,`实现依据记录.md` 也必须已经存在并能回链到本轮正式改动
- 当前编号阶段的独立归档 `reports/stage_reports/implementation_tracking/[阶段名]/实现依据记录.md` 必须存在
- 根路径 `b_class_auxiliary/runtime_checks/实现依据记录.md` 必须明确写明自己是“当前阶段副本”

### 8.2 总放行之后准备写说明文

```text
如果 `workflow_gate_report.md = pass`，再进入说明文档阶段。
请按学习型说明文规则写，不要把文档当成反向补代码质量的工具。
说明文只能整理已经存在的研究证据、实现依据证据、运行证据和质量门禁结论，不能在这里才补“当时参考了什么”。
文档风格希望通俗易懂，接近 `01_tcga_unet_project`。
如果 workflow gate 还没通过，就不要写正式说明文。
```

不要把这里理解成“到说明文阶段再补依据链”。

正确顺序只能是:

1. 研究阶段留下 `研究定标记录.md`
2. 编码阶段同步留下当前阶段 `实现依据记录.md` 独立归档，并维护根路径当前副本
3. 运行与质量阶段留下 runtime 三件套和 code quality gate
4. 总放行通过后,说明文只负责把前面已经成立的东西讲清楚

### 8.3 一轮结束后想进入下一阶段

```text
请按 skill 判断当前阶段是否已经正式结束，并告诉我下一阶段应该从哪一步开始。
如果还缺任何 formal artifact 或 gate report，就先明确指出缺口，不要直接切到下一阶段。
```

3. 当前有没有 GPU
4. 当前是否允许跑重任务

```text
当前执行环境：
```

推荐至少告诉我下面这些：

```text
当前执行环境：
- 位置：服务器
- 连接方式：`ssh user@host`
- 项目目录：`/path/to/crc_gland_segmentation_project`
```

- 现在已经在服务器上

- 服务器环境名是什么
- GPU 是否可用

### 9.3 我不会自动知道的环境事实

下面这些如果你不告诉我，我不能凭空确定：

- `sd` 里装了哪些包
- CUDA 是否能用

- 当前路径是否已经同步
- 本轮到底是 CPU smoke check 还是正式 GPU 训练

## 10. 什么时候需要租 GPU

### 10.1 不需要租 GPU 的时候

下面这些通常先在本地做：

- 治理
- 研究定标
- 阶段锁定
- Pre-check
- 工程骨架搭建
- 轻量代码修改
- CPU 级 smoke check
- 文档整理

### 10.2 建议开始考虑租 GPU 的时候

下面这些动作通常开始值得上服务器：

- `02_UNet流程验证` 进入真实训练
- 需要真实 batch / forward / loss / backward / optimizer.step 证据
- 需要较长时间训练
- 需要多次重复实验
- 需要较大输入尺寸或较大 batch

### 10.3 最推荐的切换时机

最推荐的做法不是一开始就租 GPU，而是：

1. 本地完成研究、锁定、Pre-check
2. 本地完成代码搭建和 CPU smoke check
3. 确认代码能跑、路径正确、配置基本稳定
4. 再切到服务器做正式 runtime 证据和训练

这样做的原因是：

- 本地修小问题更快
- 服务器计费更贵
- 研究和 Pre-check 不需要 GPU
- 一开始就上 GPU 容易把“制度问题”和“环境问题”混在一起
